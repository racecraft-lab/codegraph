# Feature Specification: Local HTTP Server & REST API

**Feature Branch**: `005-local-http-server`

**Created**: 2026-07-10

**Status**: Draft

**Input**: User description: "SPEC-005 — `codegraph serve --web` exposes the full graph read surface plus re-index jobs over a documented local REST API riding the existing daemon/query-pool. Local-first, dormant by default, no hosted services. Unblocks SPEC-006 (web graph browser) and reserves the SPEC-009 LSP-over-WebSocket upgrade hook."

## Overview

CodeGraph's knowledge graph is reachable today only through MCP (for agents) and the CLI (for terminal users). SPEC-006's web graph browser and SPEC-009's LSP facade need a documented local HTTP surface to build on. This feature adds one opt-in command — `codegraph serve --web` — that stands up a local REST API over the existing per-project daemon: every read capability the graph already exposes (symbol search, callers, callees, impact, graph neighborhood, status) plus repo discovery and on-demand re-index jobs, streamed as live progress. The server binds loopback by default, refuses unsafe network exposure, and adds no new runtime dependencies (design concept Q1–Q13).

The work is split into **two vertical review slices on this one branch** (Q13): Slice 1 delivers the read API end-to-end; Slice 2 layers the job subsystem on top. User stories partition cleanly across the two slices (see Reviewability Budget).

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Query the graph over a local REST API (Priority: P1) *(Slice 1)*

As a developer working in an indexed project, I run `codegraph serve --web` and query symbols, callers/callees, impact, graph neighborhoods, and server status over documented REST endpoints — served from the same warm daemon index my MCP sessions already use, so results match what my agent sees.

**Why this priority**: This is the MVP. The read API is the whole reason the server exists — it is what SPEC-006's browser renders and what a developer can exercise directly today with a browser or `curl`. Everything else hardens or extends it.

**Independent Test**: Start `codegraph serve --web` in an indexed repo, then retrieve status, a symbol search, callers, callees, impact, and a graph neighborhood over the documented endpoints and confirm each returns the expected graph data as JSON — with zero source files read by hand. Delivers a usable read API even if no other story ships.

**Acceptance Scenarios**:

1. **Given** an indexed project with a running daemon, **When** the developer starts `codegraph serve --web` and requests server status, **Then** the response reports the server version and index health so the client can confirm the API is live and which project is served.
2. **Given** the server is running, **When** the developer searches for a symbol name, **Then** the response returns matching graph nodes as JSON, drawn from the shared warm index, paged with a total count.
3. **Given** a known symbol, **When** the developer requests its callers, its callees, and its impact radius, **Then** each endpoint returns the corresponding graph relationships as JSON consistent with the same query run over MCP or the CLI.
4. **Given** a known symbol, **When** the developer requests a graph neighborhood, **Then** the response returns nodes and edges within the requested depth and signals when the result was capped.

---

### User Story 2 - Discover and address multiple indexed repos (Priority: P3) *(Slice 1)*

As a developer with several indexed projects on my machine, I list them via `/api/repos` and address any of them by repo id in repo-scoped requests; the server attaches a daemon for a non-default repo lazily, only when I first touch it (Q2).

**Why this priority**: Multi-repo turns the single-project server into the project switcher SPEC-006 wants, and `/api/repos` is in the roadmap's endpoint list. It extends US1 rather than being required for the MVP — single-repo browsing is viable without it — so it ranks below the core read API and the safe-binding hardening.

**Independent Test**: With two indexed projects registered, start the server in one, call `/api/repos`, confirm both projects are listed with the startup project marked as default, then issue a repo-scoped read against the second project and confirm its daemon is attached on demand and returns that project's data.

**Acceptance Scenarios**:

1. **Given** two indexed projects known to the daemon registry, **When** the developer requests `/api/repos`, **Then** both are listed and the project the server was started in is identified as the default.
2. **Given** a second, not-yet-attached repo appears in `/api/repos`, **When** the developer issues a repo-scoped read against it, **Then** the server attaches that repo's daemon on demand (not eagerly at startup) and returns that repo's graph data.
3. **Given** a repo id that is not indexed/registered, **When** the developer addresses it, **Then** the server returns a not-found error in the standard envelope rather than failing the request pipeline.

---

### User Story 3 - Trigger a re-index and watch live progress (Priority: P4) *(Slice 2)*

As a developer (or the SPEC-006 UI), I trigger a re-index job for a repo — incremental sync by default, or a full rebuild with `?full=true` — watch live progress over a streamed connection, get a conflict response if a job is already running for that repo, and read the last job's outcome after it finishes (Q6–Q8).

**Why this priority**: The job subsystem is the second, layered slice. It depends on the read API and repo addressing already existing, and a corrupt/partial index recovery path is valuable but not part of the minimum read surface. It ships as a separate, sequenced PR.

**Independent Test**: With the read API in place, POST a re-index for a repo, subscribe to its progress stream, observe an immediate snapshot followed by live progress and a terminal event, POST a second re-index for the same repo while the first runs and confirm it is rejected with a conflict, then read the finished job's terminal outcome.

**Acceptance Scenarios**:

1. **Given** an indexed repo with no job running, **When** the developer POSTs a re-index without `?full=true`, **Then** an incremental sync job starts; **When** they POST with `?full=true`, **Then** a full rebuild job starts instead.
2. **Given** a re-index job is running for a repo, **When** a client subscribes to its progress stream, **Then** the client receives an immediate snapshot of current state, then live progress events, and the stream ends after a terminal done/error event.
3. **Given** a re-index job is already running for a repo, **When** a second re-index is requested for the same repo, **Then** the server responds with a conflict (409) in the standard envelope and does not start a duplicate job.
4. **Given** a re-index job has finished, **When** the developer reads that repo's latest job state, **Then** the terminal outcome (succeeded or failed) is available until the server restarts.

### User Story 4 - Safe-by-default network binding and token auth (Priority: P2) *(Slice 1)*

As a security-conscious user, I trust that the server binds loopback by default and needs no auth there, refuses to start on a non-loopback host unless `CODEGRAPH_SERVER_TOKEN` is set, and then enforces that token as a Bearer header on every `/api/*` request (Q5).

**Why this priority**: Fail-closed binding is a release-blocking property of the read API, not an optional add-on — the alternative failure mode is an unauthenticated code index exposed on the LAN. It ranks just below the core read surface and above multi-repo because security-by-default cannot be deferred.

**Independent Test**: Start the server with defaults and confirm it binds loopback and serves `/api/*` without credentials; attempt to bind a non-loopback host with no token set and confirm startup is refused; set a token, bind non-loopback, and confirm `/api/*` requests without a valid `Authorization: Bearer` are rejected while valid ones succeed.

**Acceptance Scenarios**:

1. **Given** no host override, **When** the server starts, **Then** it binds the loopback interface and serves `/api/*` without requiring authentication.
2. **Given** a non-loopback host override and no `CODEGRAPH_SERVER_TOKEN` set, **When** the server is asked to start, **Then** startup is refused with a clear message and nothing binds.
3. **Given** a non-loopback host override with `CODEGRAPH_SERVER_TOKEN` set, **When** a client calls `/api/*` without a valid `Authorization: Bearer` header, **Then** the request is rejected with 401 in the standard envelope; **When** the client presents the valid Bearer token, **Then** the request succeeds.

### Edge Cases

- **Server not started with `--web`**: no HTTP listener opens, no port binds, and behavior is byte-identical to the prior release (dormancy, Constitution VII).
- **Requested port already in use / bind fails**: the server reports the failure clearly and exits without leaving a half-open listener.
- **Read against a repo whose daemon cannot be spawned/attached**: returns `error.code: "unavailable"` (HTTP 503) in the standard envelope with a `Retry-After` value, never an unhandled crash — a transient downstream-dependency failure, not a client error. *(Clarify S1-Q3.)*
- **Node id addresses a `file:`-shaped id containing slashes**: the client percent-encodes the id (`%2F` for each literal `/`) into a single path segment; the server splits the raw path on `/` before decoding, so the id round-trips correctly through the one decode site; an id that fails to decode or does not match a known node returns 404 in the standard envelope rather than fragmenting into the wrong route. *(Clarify S1-Q2.)*
- **List query exceeds the server maximum page size**: the server clamps to its hard cap rather than returning an unbounded result (Q12).
- **Graph neighborhood exceeds the node cap**: the result is truncated and the response flags it so a consumer can narrow the query (Q12).
- **Client disconnects mid-stream during a re-index**: the running job is unaffected and continues; a later subscriber re-snapshots current state (no Last-Event-ID replay, Q8).
- **Re-index job cannot acquire the index lock (external indexer holds it)**: when the daemon's file watcher or a concurrently running CLI `codegraph index`/`sync` holds the cross-process file lock the job needs, the job retries briefly (~2–3s) rather than queuing (no queue, Q8), then terminates as job `error` reason `lock_unavailable`, delivered over SSE/latest-job-state; `POST /api/reindex/:repo` itself is unaffected and still returns 202 — this is distinct from FR-022's 409, which covers only a duplicate job already tracked in this server's own registry. *(Clarify S2-Q3.)*
- **Long full rebuild starves the daemon watcher**: the daemon's file watcher exhausts its lock-retry budget after ~60s of continuous contention and permanently disables auto-sync; the job subsystem's completion/abort path must restore the watcher's sync capability so a `?full=true` rebuild doesn't silently kill live index updates for every MCP session sharing the daemon (FR-021a). *(Clarify S2-Q3, codebase-verified.)*
- **`POST /api/reindex/:repo` for a repo not in the daemon registry**: returns 404 `not_found` (`details.resource: "repo"`) per FR-010/FR-011; the API never performs first-time `codegraph init` (dormancy — Constitution VII). `?full=true` is the recovery path for an already-registered index that is corrupt or partial, not a way to initialize a new project. *(Clarify S2-Q5.)*
- **Static assets absent (all of SPEC-005's life)**: `/` serves a minimal built-in placeholder page pointing at `/api/status`, not a raw 404 (Q11).
- **Static request path attempts to escape the web root** (`GET /..%2f..%2f..%2fetc%2fpasswd` or a `..`/absolute/encoded-separator/NUL-byte path): the static file server confines every resolved path within `dist/web/` (via `validatePathWithinRoot`), reads no out-of-root file, and returns 404 `not_found` (`details.resource: "route"`) — indistinguishable from any other unmatched path, never 403 and never the file contents (FR-017b). This is the static-mount analogue of FR-004a's opaque-node-id traversal rule; the two are separate surfaces (filesystem read vs opaque DB key).
- **Unknown `/api/*` path**: returns 404 as the JSON error envelope; a missing asset-extension path (`.js`, `.css`, …) returns 404 without falling back to the app shell; only an extensionless browser route falls back to the shell (Q11, binding from SPEC-004).
- **Cross-origin request**: denied by default — the server sends no CORS headers (Q11).

## Requirements *(mandatory)*

### Functional Requirements

**Activation, process model, and dependency posture**

- **FR-001**: The system MUST expose the HTTP server only when `codegraph serve --web` is invoked; without `--web`, no HTTP listener opens, no port is bound, and observable behavior is byte-identical to the prior release (dormancy — Constitution VII; Q3). `--web` and the hidden `--mcp` mode are mutually exclusive: `serve --web --mcp` MUST fail at startup with a clear "choose one mode" error — the two are different transports with different consumers and lifecycles (stdio JSON-RPC bound to an agent's stdin vs a SIGINT-bound HTTP listener), and both already share the warm index at the daemon layer (FR-002), so co-hosting buys nothing. *(Clarify S3-Q4.)*
- **FR-002**: The serve process MUST act as a daemon *client* for all read queries — attaching to (or spawning) the per-project daemon and forwarding queries over its socket — so the web API and MCP sessions share one warm index; it MUST NOT open a second in-process copy of the index for reads (Q1).
- **FR-003**: The system MUST NOT add new runtime dependencies to satisfy this feature; the HTTP server, streaming, and static mount are built on the runtime's standard capabilities (Q4; Constitution VII). Any static-serving escape hatch is deferred until a later spec proves the need.

**Read endpoints**

- **FR-004**: The system MUST expose documented read endpoints returning JSON for: server status, symbol search, callers of a symbol, callees of a symbol, impact radius of a symbol, graph neighborhood of a symbol, and the list of indexed repos (US1, US2). The node-detail endpoint returns the node's own fields only (identity, kind, name, location, signature/doc metadata); its callers and callees are retrieved via the separate offset-paged endpoints (FR-006), and its impact radius via the separate impact endpoint, keeping the node payload bounded regardless of fan-in. The impact-radius endpoint returns a node+edge **subgraph** — the library's impact traversal yields a subgraph, not a flat list — so it shares the graph-neighborhood response shape (`{ nodes, edges, truncated }`, FR-007) and its 2000-node cap, and is deliberately NOT one of the offset-paged list endpoints (FR-006). The impact endpoint's own `?depth` defaults to 3 (the library's natural impact default) with the same maximum of 3 — it does NOT inherit the neighborhood endpoint's default of 1. *(Clarify S1-Q5; impact shape + cap ratified by consensus against `getImpactRadius → Subgraph`, which has no internal cap; 2000 matches the codebase's established dense-subgraph budget.)*
- **FR-004a**: Node ids returned by read endpoints (FR-004) are opaque tokens in one of two shapes — `<kind>:<32hex>` or `file:<relative/posix/path>`. Any endpoint that addresses a node by id in a URL path segment MUST require the client to percent-encode that id, encoding literal `/` characters in a `file:` id as `%2F`. The server MUST split the raw request path on literal `/` characters first and only then decode the matched id segment, with exactly one `decodeURIComponent` call at a single call site — the request path MUST NOT be decoded before the split. The decoded id MUST be resolved solely as an opaque database key (the existing exact-match node lookup); it MUST NOT be interpreted as a filesystem path, so a traversal-shaped id (`../`, an absolute path) is simply a key with no match, not a filesystem access. An unknown or malformed node id MUST return 404 in the standard error envelope — opaque ids make the two indistinguishable, and the API MUST NOT attempt to tell them apart. The FR-025 contract test MUST include a case addressing a `file:`-shaped id containing an encoded slash and confirm it resolves to the correct node. Operators who ever front the server with a reverse proxy MUST note that some proxies collapse or reject an encoded `%2F` in the request path; today's loopback-only default (FR-012) means this does not affect the default deployment. *(Clarify S1-Q2.)*
- **FR-005**: The status endpoint MUST report at least the server/API version and index health so a client can detect that the API is live and what it is serving, without inferring capability from other calls (Q9 — version is carried in status, not a URL prefix). The status body carries: version; the default repo's `{id, root, name}`; index summary `{state, fileCount, nodeCount, edgeCount, lastIndexed}`; hybrid-search availability and reason; and LSP availability. CLI-diagnostic internals (db size, journal mode, backend name) are excluded, and the full repo list belongs to `/api/repos`, not duplicated in status. *(Clarify S1-Q5.)*
- **FR-006**: List-style endpoints (symbol search, callers, callees) MUST support offset paging via `limit` and `offset`, defaulting `limit` to 100 with a server-enforced maximum of 500, and each response MUST carry the total count of matching items (Q12).
- **FR-006a**: The `GET /api/search` endpoint MUST accept a **required** `q` query parameter carrying the search text — an absent or empty `q` MUST be rejected with HTTP 400 and an `invalid_request` error in the standard envelope — and an optional `mode` query parameter constrained to the enum `{keyword, semantic, hybrid, auto}`, mapping 1:1 to SPEC-003's library-level search modes. The system MUST default `mode` to `auto` only when the parameter is omitted entirely. When `mode` is supplied but does not match one of the four enum values, the system MUST reject the request with HTTP 400 and an `invalid_request` error in the standard error envelope — this deliberately diverges from the MCP tool and CLI surfaces, which coerce an invalid mode to `auto`; that coercion exists to prevent agent abandonment on tool-call errors, a rationale that does not apply to REST clients, which expect and can act on a machine-readable 400. When a requested `semantic`, `hybrid`, or `auto` search degrades to keyword search (e.g., embeddings unavailable), the system MUST still return HTTP 200 with the keyword results plus `degraded: true` and a `degradationReason` field — degradation MUST NOT surface as an HTTP error. *(Clarify S1-Q4, Round 2 majority.)*
- **FR-007**: The graph-neighborhood endpoint MUST accept a traversal depth defaulting to 1 with a maximum of 3, MUST cap returned nodes at 2000, and MUST set a `truncated` flag on the response when the cap is reached (Q12).
- **FR-008**: Read results MUST be served from the shared daemon index (per FR-002), reflecting the index as of the daemon's current state (last sync / file-watch update).

**Multi-repo discovery and addressing**

- **FR-009**: The system MUST provide a `/api/repos` endpoint listing the indexed projects known to the daemon registry, with the project the server was started in identified as the default repo (Q2).
- **FR-010**: Repo-scoped endpoints MUST accept a repo identifier addressing any listed repo, and the daemon for a non-default repo MUST be attached lazily on first access rather than eagerly at startup (Q2). The repo identifier is the 16-hex-character SHA-256 prefix of the canonicalized (realpath'd) project root — the identity the daemon registry already derives as its record key. `/api/repos` returns each repo as `{ id, root, name, default }` (`name` = the root's basename; exactly one entry has `default: true` — the startup repo). The id is opaque to clients, URL-safe (hex only), and stable across server restarts. The API MUST hash the same canonical form the registry keys on (realpath), so the API id and the registry key are the same value by construction. *(Clarify S1-Q1, 2026-07-10.)*
- **FR-010a**: The repo-scoped read endpoints (`/api/search`, `/api/node/:id`, `/api/callers/:id`, `/api/callees/:id`, `/api/impact/:id`, `/api/graph/:id`) MUST accept an **optional** `repo` query parameter carrying a 16-hex repo id (FR-010) that selects the target repo; when `repo` is omitted, the read resolves against the **default (startup) repo**, so single-repo clients (US1) need no repo parameter and existing single-repo behavior is unchanged. The addressed repo's daemon MUST be attached **lazily on first access** (FR-010), never eagerly. A `repo` value that is unregistered or malformed MUST return 404 `not_found` with `details.resource: "repo"` per FR-011 (never 400). `/api/status` and `/api/repos` are deliberately NOT repo-scoped and do not accept `repo`: status reports the default repo (FR-005) and `/api/repos` lists all repos (FR-009). This query-parameter mechanism is the concrete realization of FR-010's "repo-scoped endpoints MUST accept a repo identifier" for the read surface. *(Clarify S1-Q1 follow-up, 2026-07-11.)*
- **FR-011**: Addressing a repo that is not indexed/registered MUST return `error.code: "not_found"` (HTTP 404) with `details.resource: "repo"` in the standard envelope, never an unhandled failure of the request pipeline; see FR-015a for the full code vocabulary. A repo id that is **syntactically malformed** (fails the `^[0-9a-f]{16}$` format) MUST be treated identically to an unregistered repo — 404 `not_found` (`details.resource: "repo"`), never 400 — mirroring the node-id rule (FR-004a) that a malformed and an unknown id are deliberately indistinguishable. This holds wherever a repo id is accepted: the `:repo` path segment (`/api/reindex/:repo`, `/api/reindex/:repo/events`) and the optional `repo` query parameter on read endpoints (FR-010a). *(Clarify S1-Q3.)*

**Authentication and network binding**

- **FR-012**: The default bind MUST be the loopback interface and MUST require no authentication there (Q5). The default bind host is explicitly `127.0.0.1` (never a host-less listen, which binds all interfaces) and the default port is `11235` (IANA User range; clear of the common dev-tool cluster, macOS AirPlay 5000/7000, and Neo4j 7474/7687), overridable via `--port <n>` / `--port 0` (FR-026) and `--host`. Loopback for the no-auth path is `127.0.0.0/8`, IPv6 `::1`, and `localhost`, evaluated by one shared loopback predicate (the existing embeddings-config predicate, extracted to a neutral shared util so both security gates use a single definition); the wildcards `0.0.0.0` and `::` are NOT loopback and follow FR-013's fail-closed rule regardless of the literal host argument. Additionally, the server MUST validate the `Host` header of every request against an allowlist (`localhost`, `127.0.0.1`, `[::1]`, and the bound host, each with the bound port) even on loopback binds — DNS rebinding bypasses loopback-only binding, and this server serves source code (the mitigation Vite and webpack-dev-server adopted after equivalent advisories); a request with a non-allowlisted Host is rejected with 400 `invalid_request` in the standard envelope (the FR-015a vocabulary is closed — no 403 is introduced; the offending header is named in `details`). Mutual exclusivity of `--web`/`--mcp` is FR-001. *(Clarify S3-Q1/Q5, human-approved security item, 2026-07-11.)*
- **FR-013**: Binding to any non-loopback host MUST fail at startup unless `CODEGRAPH_SERVER_TOKEN` is set — fail-closed, so an unauthenticated code index is never exposed on the network (Q5).
- **FR-014**: When a token is configured for a non-loopback bind, every `/api/*` request MUST present a matching `Authorization: Bearer` token; a missing or invalid token MUST be rejected with 401 in the standard envelope (Q5). The comparison MUST be constant-time: an empty or missing presented token is rejected before any comparison, then the SHA-256 digests of the presented and configured tokens (both UTF-8 encoded) are compared with `crypto.timingSafeEqual` — digest-first guarantees equal 32-byte inputs (never throws) and hides token length; both primitives are `node:crypto` (zero new dependency, FR-003). The token scope is every `/api/*` route including the SSE endpoint `GET /api/reindex/:repo/events`; the static mount and the placeholder `/` sit outside the auth boundary and are served without the token (public shell, authenticated data — hardened by FR-017a). Forward constraint recorded for SPEC-006/SPEC-009: the browser `EventSource` API cannot send an `Authorization` header, so a browser client of a token-bound deployment MUST use `fetch` + `ReadableStream` (or equivalent) for the SSE stream — never a token in the query string. *(Clarify S3-Q2, human-approved security item, 2026-07-11.)*
- **FR-014a**: The configured token is a secret and MUST be handled as one. Any request or diagnostic logging the server emits MUST stay local — no external egress (Constitution VII; same-origin, FR-019) — and MUST NOT record the `CODEGRAPH_SERVER_TOKEN` value, the `Authorization` request header, or the presented Bearer token, in cleartext or any reversible form. This extends the constitution's standing secret-handling rule — a credential is "never persisted, logged, or echoed" (Dogfooding §, applied there to `CODEGRAPH_EMBEDDING_API_KEY`) — to the server token. The rule binds whatever logging surface the server uses: a hand-rolled `node:http` request logger, the library `Logger` (`src/errors.ts`), or bare `console.*` MUST NOT serialize a whole request or headers object that carries the `Authorization` header verbatim. The token likewise MUST NOT appear in any response body or error envelope (already enforced for errors by the FR-015a whitelist and for the public shell by the FR-017a data-free placeholder; stated here so the no-leak property is a named requirement, not an emergent one). *(Security requirement — request-logging content rule; mirrors the human-approved FR-014 token model.)*

**Error envelope and versioning**

- **FR-015**: Every error response across the API MUST use a single envelope shape: `{ error: { code, message, details? } }` (Q9).
- **FR-015a**: `error.code` MUST be one of exactly six values: `invalid_request` (400 — malformed or negative parameters; an over-cap `limit`/`depth` MUST clamp to the maximum rather than error, echoing the effective clamped value used), `unauthorized` (401), `not_found` (404 — a single code for all not-found cases; `details.resource` MUST discriminate among `node` | `repo` | `route`), `conflict` (409), `unavailable` (503 — daemon attach/spawn failure, MUST be treated as transient and MUST carry a `Retry-After` value), `internal` (500 — unexpected faults only). `message`/`details` MUST be limited to whitelisted, schema-defined fields — never raw exception text, absolute filesystem paths, stack traces, or cause chains. `401` bodies MUST be generic and identical regardless of failure reason (enumeration prevention). *(Clarify S1-Q3; human-approved security item.)*
- **FR-016**: Endpoints MUST NOT carry a URL version prefix (no `/api/v1`); the API version is reported through the status endpoint instead, because the only sanctioned client ships in the same package as the server and cannot skew (Q9).

**Static mount and route fallback**

- **FR-017**: The system MUST serve static assets from the web build directory when present; while that directory is absent (all of SPEC-005's life), `/` MUST return a minimal built-in placeholder page that points the visitor at the status endpoint, rather than raw JSON or a bare 404 (Q11).
- **FR-017a**: The placeholder page MUST NOT embed any repo-identifying data (repo id, root path, name, or repo list) — it MUST be static and byte-identical regardless of which repos are registered, since it sits outside the `/api/*` auth boundary FR-014 defines and is reachable without a token even on non-loopback binds. *(Clarify S3-Q2.)*
- **FR-017b**: The static file server MUST confine every resolved file path within the web build directory — no URL path may resolve to a file outside `dist/web/`. The server MUST decode the request path once (bounded against multiply-encoded input) and MUST reject any path that, after decoding, escapes the root via `..`/dot-segment traversal, an absolute path, an encoded separator (`%2e%2e`, `%2f`, double-encoding), or a NUL byte; a fragile string-blocklist of `..` is insufficient (URL-/double-encoding bypasses it). Containment MUST be enforced by resolving the joined path and routing it through the repo's established content-serving chokepoint `validatePathWithinRoot` (`src/utils.ts`) — the same lexical-containment-then-symlink-aware `fs.realpathSync`-both-sides re-check the `codegraph_node`/`codegraph_explore` read sinks use to keep out-of-root file contents from leaking (#527) — which returns null on any escape. This is a surface DISTINCT from FR-004a's opaque node-id keys (which are resolved only as database keys and never touch the filesystem); the static mount does read files from disk, so it needs its own filesystem-confinement guard. A request whose path escapes the web root MUST be treated as a route miss — 404 `not_found` (`details.resource: "route"`) per FR-018, never a 403 (the FR-015a vocabulary is closed) and never the escaping file's contents — deliberately indistinguishable from any other unmatched path (no enumeration signal, mirroring FR-004a). The resolver ships in Slice 1 even though the web directory is absent for all of SPEC-005's life (FR-017), so the guard is live the moment `dist/web/` exists (a SPEC-006 dev build). The FR-025 contract test MUST include a traversal probe (e.g. `GET /..%2f..%2f..%2fetc%2fpasswd`) and confirm it returns 404 and reads no out-of-root file. *(Security requirement — static-mount path-traversal confinement.)*
- **FR-018**: Route fallback MUST follow the binding SPEC-004 rules: unknown `/api/*` paths return 404 as the JSON error envelope; requests for missing asset-extension paths (e.g., `.js`, `.css`) return 404 without falling back; only extensionless browser-route paths fall back to the app shell (Q11). A request to a **known** `/api/*` path with an **unsupported HTTP method** MUST also return 404 `not_found` with `details.resource: "route"` — the closed six-code vocabulary (FR-015a) introduces no 405, so an unsupported method on a known path is treated as a route miss rather than a distinct status.
- **FR-019**: The server MUST be same-origin only — it MUST NOT emit CORS headers and MUST NOT honor cross-origin requests (Q11; Constitution VII no-external-surface posture).

**Re-index jobs (Slice 2)**

- **FR-020**: The system MUST provide a `POST /api/reindex/:repo` endpoint that starts a re-index job for the addressed repo, defaulting to an incremental sync and escalating to a full rebuild when `?full=true` is supplied (Q6). The endpoint operates only on repos already present in the daemon registry; it never initializes a new project (Constitution VII — see Edge Cases). *(Clarify S2-Q5.)*
- **FR-021**: Re-index jobs MUST run inside the serve process via the library's sync/full-index operations, arbitrated against the daemon's file watcher by the existing cross-process file lock; the daemon MUST retain its no-indexing invariant (no new indexing RPC is added to the daemon) (Q7).
- **FR-021a**: When the re-index job's underlying sync/full-index call cannot acquire the cross-process file lock (e.g., held by the daemon's file watcher or by a concurrently running CLI `codegraph index`/`codegraph sync`), the job MUST retry acquisition for a brief bounded window (target: 2–3 seconds) rather than waiting indefinitely (no queue — Q8), then MUST transition to a terminal `error` outcome with `reason: "lock_unavailable"`, delivered over the job's SSE stream (FR-023) and reflected in the latest-job-state read (FR-024). This introduces no new HTTP status: `POST /api/reindex/:repo` still returns 202 with a job descriptor per FR-023 before lock acquisition is even attempted. The FR-022 409 response remains reserved exclusively for a duplicate active job already tracked in this server's own in-memory registry — it MUST NOT be returned for external lock contention, and lock contention MUST NOT be mapped to 503 (reserved by FR-015a for daemon attach/spawn failure). Additionally, because the shared daemon's file watcher permanently disables its auto-sync after ~60 seconds of continuous lock contention, a job that holds the lock beyond that horizon (a full rebuild) MUST NOT leave the watcher silently degraded — the job's completion or abort path MUST ensure the watcher's sync capability is restored (mechanism decided at plan time). *(Clarify S2-Q3, 2/2 consensus.)*
- **FR-022**: The system MUST allow at most one active job per repo; a re-index request for a repo that already has a job running MUST return 409 Conflict in the standard envelope and MUST NOT start a duplicate job (Q8).
- **FR-023**: Job progress MUST be delivered to subscribers as a live stream over a per-repo Server-Sent Events endpoint `GET /api/reindex/:repo/events` — a GET, because the browser `EventSource` API is GET-only; `POST /api/reindex/:repo` returns `202` with a `{ id, repo, mode, status, startedAt }` job descriptor that the client correlates to the per-repo stream (one active job per repo makes the correlation unambiguous). A subscriber receives an immediate `snapshot` event of current state on every connect (a mid-job reconnect re-snapshots; if the job already finished, the snapshot is terminal and the stream closes immediately), then live `progress` events carrying `{ phase, current, total, currentFile? }` mirroring the library's IndexProgress verbatim, and the stream ends after a single terminal `done`/`error` event; no `id:` field or Last-Event-ID replay is emitted (Q8). The stream uses `Content-Type: text/event-stream` with periodic comment heartbeats so quiet long jobs are not timed out; a client disconnect stops writes to that response but MUST NOT cancel the running job. On process shutdown with a job in flight, the server MUST abort the running job via its AbortSignal (the full-rebuild path honors the signal and releases the index lock in its cleanup path), record a terminal aborted outcome (an `error` with reason `aborted`), emit the terminal SSE event, and exit within a bounded grace period rather than deferring shutdown until the job completes; an aborted full rebuild leaves the index marked partial and recoverable by re-running the job, and a short incremental sync job may simply complete within the grace window. *(Clarify S2-Q1, S2-Q2.)*
- **FR-024**: Job state MUST be held in memory as the latest job per repo, so a client can read the last job's terminal outcome after it finishes; no job state persists across a server restart (Q8). When the addressed repo is **registered but has no job on record** (no re-index has run since the server started, or state was lost to a restart), `GET /api/reindex/:repo` and `GET /api/reindex/:repo/events` MUST return 404 `not_found` with `details.resource: "repo"` — the no-job case is deliberately indistinguishable from an unregistered repo (mirroring FR-004a's opaque-id rule), so the six-code vocabulary and its `node|repo|route` discriminator (FR-015a) are **not** extended with a separate "job" value.

**Contract honesty**

- **FR-025**: The system MUST ship a committed, hand-written API description document that documents every endpoint (path, method, and response shapes), kept honest by a contract test that walks every documented path/method/status against a running fixture server and fails on any undocumented route or mismatched response shape (Q10). "Response shapes" MUST include **every status code an endpoint can emit** under the FR-015a vocabulary — in particular the **503 `unavailable`** (with `Retry-After`) on every read endpoint that forwards to a daemon (daemon attach/spawn failure — see Edge Cases) and the **400 `invalid_request`** on every endpoint accepting client-supplied parameters (malformed/negative `q`/`limit`/`offset`/`depth`) — so the contract test walks them; an endpoint that can return a status the document omits is itself a contract failure. (A malformed node id or `repo` is the exception: it resolves to **404** `not_found`, never 400, per FR-004a/FR-011.)

**Lifecycle**

- **FR-026**: The server MUST start, serve, and stop cleanly under standard process termination, releasing the bound port on exit so a subsequent start can re-bind. On SIGINT/SIGTERM the shutdown sequence is ordered: (1) stop accepting new connections; (2) for an in-flight job, abort via its AbortSignal and emit the terminal SSE event per FR-023, and end idle SSE subscribers' responses cleanly; (3) release the bound port; (4) close every daemon client socket the serve process holds (the default repo's plus any lazily attached) — closing decrements each daemon's client refcount, and a daemon at zero clients reaps itself on its existing idle timer; the serve process MUST NOT stop or kill a daemon, because it is a client (FR-002) and a shared daemon's warm index may serve other MCP sessions; a daemon that serve itself spawned and nobody else uses idle-reaps on the same path, so nothing leaks; (5) exit within a bounded grace period. The server MUST support `--port <n>` and `--port 0` (ephemeral — the OS assigns a free port and the server prints the actual bound port, which the contract-test fixture relies on for collision-free CI). A bind failure (`EADDRINUSE`) MUST produce a clear error naming the port, suggest `--port <n>` or `--port 0`, and exit non-zero with no half-open listener. *(Clarify S3-Q3.)*

### Key Entities *(include if feature involves data)*

- **Repo (indexed project)**: an entry in the daemon registry addressable by a repo identifier; carries its project root and index/health status, and one entry is flagged as the server's default. Shape: `{ id, root, name, default }` where `id` is the 16-hex SHA-256 prefix of the realpath'd root — the registry's own record key (FR-010).
- **Read query result**: graph nodes and/or edges returned as JSON for a read endpoint; list results additionally carry `total`, and echo the effective paging window; the graph-neighborhood result additionally carries `truncated`.
- **Re-index job**: a unit of re-index work for one repo; carries its mode (incremental sync or full rebuild), lifecycle status, progress information, and start/finish timing. In memory only, latest-per-repo, lost on restart. Job id is a `crypto.randomUUID()`; lifecycle is `running` → terminal `done` | `error` (no queued state — a job is created running; an aborted shutdown is an `error` with reason `aborted`; a job that cannot acquire the underlying index lock within its bounded retry window is an `error` with reason `lock_unavailable` — the `reason` values are an open-but-documented list: each new value gets spec'd, mirroring FR-015a's discipline); progress carries the library's IndexProgress phase enum `scanning | parsing | storing | resolving | embedding` verbatim (`embedding` appears only when embeddings are configured); the terminal `result` is documented per mode, because the library's incremental-sync and full-index result shapes differ. *(Clarify S2-Q4.)*
- **Progress event**: a message on a job's live stream — named events `snapshot` (full job descriptor, sent on every subscribe), `progress` (`{ phase, current, total, currentFile? }`), and a single terminal `done`/`error` event that ends the stream. *(Clarify S2-Q1.)*
- **Error envelope**: the uniform error shape `{ error: { code, message, details? } }` returned by every failing request.
- **Server status**: the descriptor returned by the status endpoint — at least API/server version and index health, sufficient for a client to confirm liveness and served project.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: A developer can start `codegraph serve --web` in an indexed repo and retrieve status, symbol-search, callers, callees, impact, and graph-neighborhood results over the documented endpoints — every read capability available via MCP/CLI is reachable over HTTP — without reading any source file by hand.
- **SC-002**: Safe-by-default binding holds 100% of the time: with defaults the server binds loopback and serves `/api/*` without credentials, and an attempt to bind a non-loopback host with no token set prevents startup every time (nothing binds).
- **SC-003**: With a token configured on a non-loopback bind, 100% of `/api/*` requests lacking a valid Bearer token are rejected and 100% of requests with the valid token succeed.
- **SC-004**: A re-index triggered via the API surfaces live progress to a connected client and a terminal outcome the client can read after completion; a duplicate re-index for the same repo is rejected while one is running, 100% of the time.
- **SC-005**: The committed API description matches the running server — the contract test detects any undocumented route or mismatched response shape, and CI tolerates zero undocumented endpoints.
- **SC-006**: With `--web` not passed, the tool performs zero network binds and behaves identically to the prior release (dormancy verified).
- **SC-007**: Each of the two review slices lands as its own PR under the ~400 reviewable-LOC ceiling.
- **SC-008**: The API is exercised against this repository's own index as a self-repo dogfood step — `serve --web` here answers `/api/status` and a symbol search with this project's data (Constitution "Dogfooding (binding)").

## Out of Scope

- The web UI itself — delivered by SPEC-006; SPEC-005 ships only the API and a placeholder page.
- The LSP-over-WebSocket handler — SPEC-009; SPEC-005 reserves the connection-upgrade path but implements no WebSocket or LSP behavior (nothing to assert here, so it is a boundary note, not a functional requirement).
- TLS / HTTPS termination — reverse-proxy territory.
- Any new daemon RPC for indexing — the daemon keeps its no-indexing invariant; jobs run in the serve process (Q7).
- Job history or persistence — only the latest job per repo is kept, in memory, lost on restart (Q8).
- CORS / cross-origin access — same-origin only (Q11).
- Cursor-based pagination — offset/limit with hard caps only (Q12).
- URL versioning (`/api/v1`) — version is reported in status (Q9).
- Eager attach of all registered repos — lazy attach only (Q2).

## Assumptions

- The per-project daemon, its registry, and the attach-or-spawn machinery the MCP proxy uses already exist and are reused unchanged for read queries (Q1, Q2).
- The existing cross-process file lock is sufficient to arbitrate an in-process re-index job against the daemon's file watcher, and the library's sync/full-index operations expose progress suitable for driving the live stream (Q7).
- On a loopback bind, no authentication is required even if `CODEGRAPH_SERVER_TOKEN` happens to be set — the Bearer requirement is a property of non-loopback binds (design-concept reading of Q5; the precise loopback-plus-token interaction is folded into the bind/auth clarification).
- The web build directory is absent for all of SPEC-005's lifetime, so the placeholder page is the only HTML served (Q11).
- Read endpoints reflect the daemon's current index state; correcting for a stale index mid-read is a UI concern (SPEC-006), not handled here.
- The reviewable-LOC projection (~620) uses the roadmap's recorded estimator numbers; the plugin's live size-estimator operation is not installed in this environment (design concept Open Questions), which does not affect the accepted 2-slice decision.
- No TLS: operators who need network exposure put the loopback server behind their own reverse proxy.
