# Error-Handling Checklist: Hybrid Semantic Search

**Purpose**: Requirements-quality validation ("unit tests for English") for the
error-handling / graceful-degradation requirements of SPEC-003 — every fallback
path's success-shaped response, the never-error posture (Constitution VI), provider
lifecycle/latching under failure, and integrity of cache/provenance state when the
embed budget expires while an embed is still in flight.
**Created**: 2026-07-09
**Feature**: [spec.md](../spec.md) · [plan.md](../plan.md) · [contracts/degradation-hints.md](../contracts/degradation-hints.md)

**Note**: Generated by `/speckit-checklist error-handling`. Each item tests whether an
error-handling requirement is present, clear, measurable, consistent, and complete —
NOT whether the implementation handles the error correctly.

- [ ] CHK002 Is the model-mismatch condition specified as folding into an existing hint (string 2) rather than being a separate or undefined fallback path? [Consistency, Spec §Edge Cases/§FR-015]
- [ ] CHK003 Is the large-index memory-guard skip specified as folding into an existing hint (string 4) rather than introducing a fifth undefined condition? [Consistency, Spec §FR-009c]
- [ ] CHK004 Are the "no vectors" and "no provider" conditions kept as distinct hint strings (2 vs 1) rather than conflated into one fallback message? [Clarity, Spec §FR-015 strings 1/2]
- [ ] CHK005 Does every degraded condition trace to a single source-of-truth string set shared by spec.md and the degradation-hints contract? [Traceability, Spec §FR-015 · contract]

## Provider Lifecycle & Failure Latching

- [ ] CHK010 Is lazy provider initialization on the first hybrid-eligible query specified, with that first query served keyword-only plus the warming note? [Completeness, Spec §FR-005]
- [x] CHK011 Is provider-init FAILURE (a lazy init that throws/never becomes ready, distinct from a per-query embed timeout) given defined latch semantics — whether subsequent queries retry init or stay keyword — so a failed provider neither wedges the daemon nor re-blocks every subsequent query for the full embed budget? [Resolved →FR-005] (failed init → keyword + string 4, provider returns uninitialized, later query MAY re-attempt, serialized behind single owner and bounded by embed budget; mirrors existing anti-latch provider posture)
- [ ] CHK012 Is the warming→ready transition specified so later queries fuse once ready, and is the warming note required on every query while warming (not one-shot)? [Clarity, Spec §FR-005]
- [ ] CHK013 Is concurrent first-query behavior during a cold cache defined (serialized single-owner build, no thundering-herd redundant builds)? [Coverage, Spec §FR-009b]

## Embed-Budget Expiry & Late-Vector Integrity

- [ ] CHK014 Is the per-query embed wait bounded by a hard cap so no query blocks past ~2 s before falling back to keyword + hint? [Clarity, Spec §FR-006/§FR-006a]
- [x] CHK015 Is the disposition of an embed that COMPLETES AFTER the budget expired specified — the late-arriving query vector MUST NOT mutate the matrix cache, corrupt provenance, or alter the already-returned keyword response? [Resolved →FR-006] (late vector discarded; cache holds only corpus vectors never query vectors; returned keyword result is final; contract mirrors)
- [ ] CHK016 Is the degraded-path latency ceiling stated (keyword latency + at most the embed budget) and required to be verifiable as an elapsed-time bound, not only via the keyword-fallback output? [Measurability, Spec §FR-006a]
- [ ] CHK017 Is a budget-expired query required to render hint string 4 and remain success-shaped (no `isError`)? [Consistency, Spec §FR-006/§FR-015 string 4]
- [x] CHK018 Is a catch-all defined for any unexpected exception on the semantic/hybrid path (e.g. the staleness-probe read, matrix decode, or fusion step throwing) so it degrades to keyword + hint string 4 and never surfaces as `isError`? [Resolved →FR-006] (string 4 designated the catch-all for every non-happy-path semantic outcome that is not condition 1/2/3; contract mirrors)
