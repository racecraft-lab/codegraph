# Security Requirements Quality Checklist: Bundled Local Embedding Fallback

**Purpose**: Adversarial validation of the download-trust + path-traversal requirement surface (SPEC-002) — the dedicated Phase-4 security pass deferred from Clarify Session 3. Tests whether the SECURITY REQUIREMENTS themselves (in spec.md / plan.md / contracts / data-model) are complete, clear, consistent, and measurable — NOT whether code works.
**Created**: 2026-07-05
**Feature**: [spec.md](../spec.md) · [plan.md](../plan.md) · [contracts/model-fetch.md](../contracts/model-fetch.md) · [data-model.md](../data-model.md) · [research.md](../research.md)
**Threat framing**: the in-source SHA-256 pin is the SOLE trust anchor; the download host (HF hub AND any `CODEGRAPH_MODEL_BASE_URL` override) is UNTRUSTED. Adversary goals assessed: inject unverified bytes, bypass verify-before-use, escape the cache root, turn the override into an SSRF/ACE/exfil primitive, exhaust local resources, and leak secrets in messages.

## Trust Anchor & Checksum Completeness

- [x] CHK001 Is the SHA-256 pin's storage location (compiled into CodeGraph source) and its status as the SOLE trust anchor for an untrusted host explicitly specified? [Completeness, Spec §FR-013 / Assumptions]
- [x] CHK002 Is it stated unambiguously that the same pinned verification applies to bytes obtained from ANY source — the default hub AND the `CODEGRAPH_MODEL_BASE_URL` override — so the override host cannot inject unverified bytes? [Clarity, Spec §FR-015 / contract §6]
- [x] CHK003 Is checksum verification required as a NORMATIVE requirement (FR-level) for EVERY downloaded artifact — including `tokenizer.json` — rather than only the `.onnx` model? FR-013/FR-014/SC-003 named only "the model"/"the downloaded bytes" (singular); the tokenizer's verification appeared only in Clarifications/research prose, so a reader implementing the FRs literally could load an unverified tokenizer from the untrusted host. (closed by FR-013/FR-014 rewrite — "EACH downloaded artifact... the quantized model AND the tokenizer file")
- [x] CHK004 Is the pinned SHA-256 VALUE actually recorded for every artifact the loader verifies, with no pin left as an implement-time TODO? (research.md OQ-2 and data-model §3 leave `tokenizer.json`'s digest "pinned at implement time" — the trust anchor is incomplete.) [Completeness, Spec §research OQ-2 / data-model §3]
- [x] CHK005 Are the artifacts named for download internally consistent across artifacts? (Clarifications says "tokenizer.json/vocab.txt"; research OQ-2 / data-model / contract say only `tokenizer.json` with vocab embedded.) [Consistency, Spec §Clarifications vs §research OQ-2]
- [x] CHK006 Is the "0% unverified bytes used" success criterion (SC-003) scoped to cover ALL verified artifacts, or only the model? (SC-003 reads "A downloaded MODEL whose bytes do not match…" — measurably model-only.) [Measurability, Spec §SC-003]

## Verify-Before-Use & Atomic Acquisition

- [x] CHK007 Is "verify BEFORE use" specified strongly enough that unverified bytes are never opened/parsed by the ONNX runtime (the parser is itself attack surface), not merely "not used for the final embedding"? [Clarity, Spec §FR-014 / contract §Behavior 4-5]
- [x] CHK008 Are requirements defined so a partial/interrupted download is treated as absent and never used as if complete (atomic verify-then-rename; a present `path` is always complete+verified)? [Coverage, Edge Case, Spec §Edge Cases / FR-018 / data-model §3]
- [x] CHK009 Is re-verification of an already-cached artifact on every reuse required (anti cache-poisoning), rather than a bare existence check? (Contract step 2 requires an sha256 match on reuse; FR-018's "reuse without re-downloading" wording alone is looser and should carry the re-verify property.) [Consistency, Spec §FR-018 / contract §Behavior 2]
- [x] CHK010 Is the missing/corrupt-`.wasm` infinite-hang gotcha bounded by a requirement (timeout-wrapped session init) so a "model unavailable" case degrades instead of freezing the index? [Coverage, Spec §research OQ-1 / contract local-provider / plan Constraints]

## Untrusted-Host Resource & SSRF Bounds

- [x] CHK011 Is there a requirement bounding the bytes read (a size cap, using the pinned exact byte lengths) AND a download wall-clock timeout, so a hostile/MITM host cannot cause disk-exhaustion via an unbounded stream or a slow-loris hang? FR-019 covers a "network failure" (offline), but an unbounded/slow stream is neither offline nor a checksum mismatch and was previously unspecified. (closed by FR-013a)
- [x] CHK012 Are the `CODEGRAPH_MODEL_BASE_URL` override's URL SCHEME (constrained to http/https) and trust boundary (env-sourced, operator-controlled) specified, so the request-side SSRF/exfil surface is addressed? The pinned checksum neutralizes the returned BYTES but never prevents the outbound REQUEST itself (e.g. to `http://169.254.169.254/…` or a `file:`/other-scheme target); FR-015 previously constrained only "base vs full-file-URL," not scheme or provenance. (closed by FR-015 append — http/https scheme allowlist + env-only provenance; also see Clarifications "Override SSRF scope" decision)
- [x] CHK013 Is the integrity property — that the pinned checksum makes host tampering (default or override) unable to inject usable bytes REGARDLESS of the override URL — stated as a requirement? [Clarity, Spec §FR-015 / contract §Security surface]

## Cache-Directory Path Validation (Traversal / Sensitive Paths)

- [x] CHK014 Are the cache-path rejection SEMANTICS specified precisely — prefix-match (not exact-match) for sensitive system roots so subdirectories of `/etc`, `/usr`, `/var`, … are rejected for a WRITE sink, and symlink-resolved (realpath) evaluation so a symlinked cache dir cannot escape? The reused guard's `SENSITIVE_PATHS` check in `src/utils.ts` is EXACT-match (catches `/etc`, misses `/etc/cg`), and the purpose-built cache validator's symlink behavior was undefined; FR-017a previously said "names a sensitive system location" without pinning either semantic. (closed by FR-017a rewrite — PREFIX match, evaluated after realpath)
- [x] CHK015 Is the false-reject constraint specified — the cache validator MUST accept a legitimate `XDG_CACHE_HOME` under `~/.config` (i.e. NOT reuse `validateProjectPath` verbatim, which rejects `.config`)? [Clarity, Spec §FR-017a / research finding 1]
- [x] CHK016 Is safe temp-file creation in the shared machine-wide cache required — exclusive create (no clobber), non-symlink-following, and an unpredictable name — so a pre-planted symlink or a predictable temp name cannot redirect the write or win a race? The "shared cache under concurrency" edge case covered non-corruption but not the symlink/predictable-name write-safety threat (a concern the repo already tests for elsewhere). (closed by FR-017a rewrite — exclusive create, no symlink-follow, unpredictable name)
- [x] CHK017 Are requirements defined so a rejected/unwritable cache directory (including a bad `CODEGRAPH_MODEL_CACHE_DIR`) degrades like the offline case — actionable message, structural index still completes? [Coverage, Edge Case, Spec §Edge Cases / FR-017a]

## Secret / URL Redaction & Transport Security

FR-019/FR-019a previously said messages "name `CODEGRAPH_MODEL_BASE_URL`" without specifying that its value is redacted; SPEC-001 established this precedent (FR-023) precisely because URLs carry credentials. (closed by FR-019/FR-019a append — redact to scheme+host+port via `redactEndpoint`)

(closed by the new Clarifications "Plaintext-http override" decision bullet)
- [x] CHK020 Is it required that no source text or composed model input is echoed into any failure/unavailability reason, so error paths do not leak indexed code? (now normative via FR-019c) [Consistency, Spec §contract local-provider §Lifecycle]

## Notes

- **Adversarial verdict**: The core trust model is SOUND — a pinned in-source SHA-256 with atomic temp→verify→rename genuinely neutralizes host/MITM byte-injection regardless of the `CODEGRAPH_MODEL_BASE_URL` override (a mismatched or partial file is discarded, never renamed into place, never opened by the ONNX runtime). No bypass of the checksum anchor was found. The gaps are at the EDGES of that anchor: a second untrusted artifact (tokenizer) not held to the same normative bar, unbounded resource/request behavior from the untrusted host, imprecise cache-write rejection semantics, and secret-redaction/transport parity with SPEC-001 that was not carried forward.
- **Plaintext-http assessment (CHK019)**: Parity with SPEC-001's `plaintextRemoteWarning` is materially WEAKER here and is assessed as NOT strictly required on its original (confidentiality) grounds — the model is a PUBLIC artifact (no source code or bearer key crosses the wire, unlike the SPEC-001 endpoint), and the pinned checksum removes the integrity risk (injected bytes fail verification). The only residual is (a) a MITM forcing repeated checksum-mismatch, already handled as the FR-019a degrade, and (b) credential leak if an override carries userinfo over http — which the CHK018 redaction fix covers. Recommendation: record this assessment + decision in the spec; a one-line advisory is OPTIONAL, not mandatory. This is the item most appropriate for consensus ratification.
- **Remediation handling**: Per the parallel-batch constraint, spec.md/plan.md were NOT edited by this executor. Exact proposed remediation text for every unchecked gap item is reported to the parent for serial application; verification (re-run + marker recount) occurs after the parent applies the edits.
- Items marked `[x]` = the requirement is present and adequately specified (traceability cited). Items left unchecked and tagged as a gap = a genuine missing/under-specified security requirement requiring a spec/plan edit.
