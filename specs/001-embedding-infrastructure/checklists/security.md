# Security Checklist: Embedding Infrastructure & Endpoint Provider (SPEC-001)

**Purpose**: Validate the *quality* (completeness, clarity, consistency, measurability, coverage) of the **security requirements** in `spec.md` + `plan.md` + `contracts/` — API-key confidentiality (env-only, never persisted/logged/echoed), endpoint-URL / embedded-credential safety, the code-content egress/exfiltration scope (source code leaves only to the user-configured endpoint), zero-telemetry / no-phone-home, and credential safety on every error/log sink. This is "unit tests for the requirements," not the implementation.
**Created**: 2026-07-04
**Feature**: [spec.md](../spec.md) · [plan.md](../plan.md) · [research.md](../research.md) · [contracts/](../contracts/)
**Domain focus**: `CODEGRAPH_EMBEDDING_API_KEY` env-only, never written to `.codegraph/` / config / logs / errors / status · endpoint URL = user infrastructure (no telemetry, no phone-home, Q9 zero new telemetry) · embedding inputs (code snippets) leave the machine ONLY to the user-configured endpoint, only when explicitly configured · **special attention** — failed-HTTP-call error messages must not echo the `Authorization` header or the full request body
**Status**: Loop 2 — all 4 gaps surfaced in Loop 1 remediated in spec.md/plan.md (see `[Resolved →§…]` tags): added FR-025a (code-egress + local-sink confidentiality) and SC-011 (measurable single-destination/no-leak criterion), extended FR-023 (URL-credential at-rest non-persistence), and added the Assumptions "Transport confidentiality" scope bullet; plan.md Constraints mirror all four. All four gaps carry security keywords (credential / api-key / code exfiltration), so each is flagged for defense-in-depth consensus per the security-keyword rule even though remediated with high confidence.

## API Key Confidentiality (env-only lifecycle)

- [ ] CHK001 Is it required that the API key is sourced ONLY from `CODEGRAPH_EMBEDDING_API_KEY` (environment), held in memory only, and never written to `.codegraph/`, a config file, or any other persisted artifact? [Completeness, Spec §FR-023, contract embedding-config §Redaction]
- [ ] CHK002 Is it required that the API key never appears in any log line, error message, or `status` / `--json` output across ALL success and failure paths (auth failure, 5xx body echo, timeout, dimension conflict, malformed response)? [Completeness, Spec §FR-023/SC-007]
- [ ] CHK003 Is the key-confidentiality guarantee required to hold identically whether the pass runs under the CLI or the background daemon's watcher-triggered sync, across every named sink (terminal output, the persisted error log, the daemon log)? [Consistency, Spec §FR-023/SC-007/FR-015]
- [ ] CHK004 Is the key required to be transmitted solely as the `Authorization: Bearer` header, omitted entirely for keyless endpoints (so a keyless local endpoint is never sent — and no code path fabricates — an empty or placeholder credential)? [Coverage, Spec §FR-003, contract embedding-provider §2]
- [ ] CHK005 Is provider response-body content treated as untrusted for key leakage — a response body that echoes a real fragment of the submitted key must pass the same redaction before any part of it is surfaced? [Coverage, Spec §FR-023]

## Endpoint URL & Embedded-Credential Safety

- [ ] CHK006 Is the endpoint required to be rendered redacted to **scheme + host + port only** (userinfo, path, and query stripped) in EVERY rendering — the `status` human section, the `--json` `embedding.endpoint`, and any error/log line? [Completeness, Spec §FR-023/FR-022, contract status-embedding-json §endpoint]
- [ ] CHK007 Is redaction required to survive an unparseable `CODEGRAPH_EMBEDDING_URL` — rendering a safe non-credential placeholder (never the raw string) and treating the unparseable URL as an advisory endpoint failure rather than a crash that could print the raw value? [Edge Case, Spec §FR-023]
- [x] CHK008 Is it required that the endpoint URL — and any credentials embedded in it (userinfo / query parameters) — are never **persisted** to `.codegraph/` (neither the vector store nor `project_metadata`, which holds only the model name and dimension), parallel to the key's non-persistence guarantee — i.e., the at-rest non-persistence of URL-embedded credentials, not only their non-rendering? [Resolved →§FR-023: extended so URL-embedded credentials (userinfo/query) are never **persisted** under `.codegraph/` — neither `node_vectors` nor `project_metadata` (which holds only model + dims, FR-004/FR-009); URL + key are memory-only — parallel to the key's non-persistence, not merely never rendered; plan Constraints mirror it]
- [ ] CHK009 Is the redaction guarantee required to extend recursively through an error's `cause` / wrapped-error chain and to non-message own-properties (e.g. the `.input` string Node attaches to an invalid-URL error), enforced by replacing (never merely wrapping) the underlying transport error at the throw site? [Coverage, Spec §FR-023/SC-007]

## Code/Input Egress & Exfiltration Scope (special attention — what leaves the machine)

- [x] CHK010 Is there an AFFIRMATIVE requirement that the composed embedding input (which contains the user's source code) is transmitted SOLELY to the user-configured endpoint, ONLY while the feature is active, carrying ONLY the standard embeddings payload (model + input) with no added identifier/path/metadata, and that the feature introduces no secondary destination, default/fallback endpoint, analytics, or phone-home channel? The property is currently only implied by the union of dormancy (FR-002) + activation (FR-001) + no-new-telemetry (FR-025), never stated affirmatively as a code-egress invariant. [Resolved →§FR-025a + §SC-011: added FR-025a — the composed input (source code) leaves the machine ONLY as the `{model, input}` request body to the ONE user-configured endpoint, only when active, with no added metadata, no secondary/fallback destination, no phone-home, and no embedding-derived data added to the existing anonymous telemetry; SC-011 makes it measurable (single destination + exact payload shape, CLI/daemon parity); plan Constraints mirror it]
- [x] CHK011 Is it required that the composed embedding input (source-code content) is never written to any log line, error message, or persisted artifact — only its SHA-256 `input_hash` is persisted — so proprietary code cannot leak into a local sink (terminal, errors.log, daemon.log) independent of the outgoing-request-object protection (e.g. a per-symbol failure names the node id, never echoes its source text)? [Resolved →§FR-025a + §SC-011: FR-025a's local-sink clause — the composed input is written to no log line, error message, or persisted artifact beyond its SHA-256 `input_hash` (FR-008); a per-symbol failure names the node id, never the source text; explicitly additional to FR-023's request-object protection; CLI + daemon parity (FR-015); SC-011 mirrors]
- [ ] CHK012 Is the outgoing request/init object — which carries BOTH the `Authorization` header AND the request body (the code content) — required to never appear in any logged, dumped, or error-context representation? [Completeness, Spec §FR-023]
- [ ] CHK013 Is fully-unconfigured dormancy required to make ZERO embedding-endpoint network requests (no source code leaves the machine at all when the feature is off), byte-identical to a build without the feature? [Coverage, Spec §FR-002/SC-002]

## Telemetry & Phone-Home (Q9: zero new telemetry)

- [ ] CHK014 Is it required that the feature introduces no new runtime dependency and emits no telemetry event (Q9), so enabling embeddings adds no outbound analytics traffic? [Completeness, Spec §FR-025/SC-008]
- [ ] CHK015 Is it required that no embedding-specific data — the endpoint URL/host, the model name, code content, coverage figures, or the fact that an embedding pass ran — is added to the existing anonymous usage telemetry (which collects only language names + coarse buckets, never paths/code/endpoints), given the pass runs inside the already-instrumented index/sync? [Consistency, Spec §FR-025]

## Error/Log Sink Credential Safety (special attention — failed HTTP calls)

- [ ] CHK016 Is redaction required on error/log messages **derived from underlying transport failures** (raw `fetch`/network error objects, endpoint response bodies), re-rendered with the endpoint redacted rather than surfaced verbatim, on EVERY failure path (timeout, network error, 4xx/5xx, dimension conflict, malformed response)? [Coverage, Spec §FR-023]
- [ ] CHK017 Are the output sinks the redaction/credential-safety rules apply to explicitly enumerated (terminal output, the persisted error log, the daemon's log) and required to behave identically under CLI vs. daemon-watcher execution? [Clarity, Spec §FR-023/SC-007]

## Transport Confidentiality

- [x] CHK018 Is the transport-confidentiality posture specified — that the endpoint scheme (`http` vs `https`) is user-controlled, that CodeGraph transmits to exactly the configured URL without upgrading or downgrading TLS, that keyless/plaintext **local** endpoints are supported by design, and that confidentiality of the code payload (and any bearer key) sent to a **remote** endpoint over plaintext `http` is the user's responsibility — so the spec states whether plaintext-`http` egress is permitted and its security implication rather than leaving it undefined? [Resolved →§Assumptions "Transport confidentiality": the endpoint scheme is user-controlled; CodeGraph transmits to exactly the configured URL, neither upgrading nor downgrading TLS; plaintext-`http` **local** endpoints supported by design (US1 AS3); `https` for a **remote** endpoint is the user's responsibility; a plaintext remote URL is not blocked/warned this version (deferred hardening); does not weaken FR-023; plan Constraints mirror it]

## Trust Boundary, Threat Model & Scope

- [ ] CHK019 Is the trust boundary documented — env-sourced config is trusted input; the configured endpoint is semi-trusted (its response body is untrusted per FR-021a/FR-023); the user's own endpoint is not modeled as an adversary (so response-size/DoS from the user's own endpoint is out of scope)? [Assumption, Spec §FR-021a/FR-023]
- [ ] CHK020 Is same-OS-user process-environment visibility of the key (e.g. `/proc/<pid>/environ`, `ps e`) acknowledged as inherent to env-var configuration and out of this feature's threat model, rather than left implying an unaddressed leak? [Assumption, Spec §FR-023]
- [ ] CHK021 Is the half-configuration actionable error (FR-001a) required to name only the MISSING variable and never echo the VALUE of the set variable (which may embed URL credentials), consistent with FR-023's "never rendered in any output"? [Consistency, Spec §FR-001a/FR-023]
- [ ] CHK022 Are the security success criteria objectively measurable — SC-007 (key never in any persisted file / log / error across all paths, including the `cause` chain and daemon parity) and SC-008 (zero telemetry) — so the guarantees can be verified rather than asserted? [Measurability, Spec §SC-007/SC-008]
- [ ] CHK023 Is it required that the feature adds no new attack surface to the retrieval/MCP tool surface (vectors are written but never consumed here; no new agent-facing input path is introduced)? [Coverage, Spec §FR-026]

- Marker legend: a "Gap" tag marks a **security** requirement that is missing, under-specified, or internally inconsistent in `spec.md`/`plan.md`/`contracts/`. Items citing `[Spec §…]`/`[contract …]` without a "Gap" tag reference an existing, adequately-specified requirement whose quality this item audits (the security surface was already hardened by FR-023 — extended twice — SC-007, FR-025, FR-001a, and three consensus rounds; those items confirm coverage rather than re-open it). A `[Resolved →§…]` tag with a checked box marks a Loop-1 gap now closed by a spec/plan edit.
- Loop-1 gaps (4): CHK008 (URL-credential at-rest non-persistence), CHK010 (affirmative code-egress invariant), CHK011 (input-content local-sink confidentiality), CHK018 (transport-confidentiality posture). All four carry security keywords (credential / api-key / code exfiltration), so each is flagged for defense-in-depth consensus per the security-keyword rule even after remediation.
- Grounding: the OpenAI-compatible request body is exactly `{ model, input[] }` with an optional `Authorization: Bearer` header and no vendor fields (`contracts/embedding-provider.md` §2). The existing anonymous telemetry collects only language names + coarse file-count/duration buckets — "never paths, file names, or exact counts," and never the endpoint or code content (`src/telemetry/index.ts`). `project_metadata` persists only `embedding_dims` + `embedding_model` (D9, `contracts/node-vectors-schema.md`), never the endpoint URL. No URL-redaction/credential-scrub helper exists in `src/` today (verified: only an unrelated `/^sanitize/i` framework-matcher in `src/resolution/frameworks/express.ts`), so FR-023's redaction and the code-egress invariant are net-new spec requirements, not inherited behavior. The feature is unimplemented (`CODEGRAPH_EMBEDDING*` appears nowhere in `src/`), so these are pure requirements-quality audits.
- Overlap boundaries (deliberate, declared): `error-handling.md` owns the *failure-semantics* lens on credential safety (advisory-abort mechanics, retry/timeout, and CHK021/CHK022 there — key-never-in-error + error-path redaction); this checklist audits the same rules through the *credential/code-confidentiality invariant* lens across every sink, mirroring how `performance.md` re-examines the retry constants only through the wall-clock lens. `data-integrity.md` owns persistence/hashing/reconciliation correctness; this checklist's persistence items (CHK008) audit only the *confidentiality* of what is persisted, not its integrity.
