- [x] CHK002 Does the spec honestly scope the "cost proportional to changed-node count" claim to **endpoint work** (calls / vectors written), given the per-sync staleness selection recomposes+rehashes across the embeddable set and the FR-017 anti-join scans the whole live-node set — i.e., is the fixed, network-free `O(embeddable-symbols)` per-sync overhead acknowledged and bounded rather than implied to be `O(changed)`? [Resolved →§FR-027 + §SC-003: added FR-027 scoping the incremental guarantee to endpoint work (requests/writes ∝ changed/missing set), naming the staleness scan + FR-017 anti-join as network-free `O(embeddable-symbols)` local work that issues no request or write, and noting the scan is deliberately not narrowed to changed-file scope because FR-020 resume needs the wider scan; SC-003 amended to say the proportionality guarantee is on endpoint work; plan Performance Goals mirrors]
- [ ] CHK003 Is a daemon-watcher single-file edit required to re-embed only the changed file's symbols (a handful → ≤1–few batched HTTP calls), not the whole project? [Coverage, Spec §FR-015/FR-016, US2 AS4]
- [ ] CHK004 Is the observable signal for "re-embed count == changed inputs" defined (the endpoint request count), so proportionality is objectively measurable? [Measurability, Spec §SC-003, quickstart §B1]
- [ ] CHK005 Is the daemon-watcher path required to run the embed pass identically to a CLI sync with no separate wiring (the pass lives in `sync()`)? [Consistency, Spec §FR-015, research §D13]

## Full-Index Scale, Memory & Progress

- [x] CHK006 Is the embed pass required to select and process eligible nodes in **bounded batches** (streaming) rather than loading all embeddable symbols and their ~6,000-char composed inputs into memory at once — following the "batched resolution to avoid OOM" precedent (`src/index.ts`)? [Resolved →§FR-028: pass MUST select/process eligible symbols in bounded batches (streaming) and MUST NOT materialize all embeddable symbols + composed inputs at once, keeping peak memory bounded independent of repo size; grounded in the existing batched-selection-to-avoid-OOM precedent; plan Performance Goals mirrors]
- [ ] CHK007 Are progress requirements defined for the embed pass on a long full index (an `embedding` phase emitted only when active; progress = embedded ÷ eligible) so a 10k+-file pass reports advancement? [Completeness, Spec §FR-022, research §D15]
- [x] CHK008 Is there a measurable performance target (throughput or a wall-clock bound) for a full-index embed on a large (10k+-file) repo — or is the deliberate **absence** of one explicitly justified (embed wall-clock is endpoint-bound, not codegraph-bound; the codegraph-side bound is batching + bounded concurrency + per-request timeout + bounded retry)? [Resolved →§SC-010: added SC-010 — no fixed wall-clock target because throughput is endpoint-bound, not codegraph-bound — enumerating the measurable codegraph-side bounds as the guarantees (batch/concurrency/timeout/retry, bounded memory FR-028, batch-txn writes + WAL checkpoint FR-029/FR-030, responsive reads FR-031, observable `embedding` progress phase) plus the failure bound (abort within one batch's retry budget); plan Performance Goals mirrors]
- [ ] CHK009 Is the per-row vector write volume bounded and predictable (`byteLength == dims * 4`), so total bulk-write size on a full index is a known function of symbol-count × dims? [Clarity, contract node-vectors-schema, data-model rule 3]

## Vector Write Batching & Transaction Churn

- [x] CHK010 Is it required that vector persistence is committed in **batch-sized transactions** (not one implicit transaction per `upsertNodeVector` row), avoiding per-row fsync churn, following the `db.transaction()` bulk-write precedent (`src/db/queries.ts`)? [Resolved →§FR-029: vectors of one completed batch committed together in a single transaction following the existing bulk-write precedent, never one implicit transaction per row (per-row fsync churn); node-vectors-schema contract "Write batching & WAL checkpoint" note added]
- [x] CHK011 Is it required that the pass does **not** hold a single transaction open across the whole pass (spanning HTTP awaits) — which would bloat the WAL and make partial progress non-durable, contradicting resume-from-partial-progress — so each committed batch is durable and an aborted pass leaves already-written vectors committed on disk? [Resolved →§FR-029: MUST NOT hold a pass-long transaction across network I/O; each batch commits before the next begins, so already-written vectors are durably on disk when a later batch aborts — the FR-019/FR-020 resume/backfill contract explicitly depends on this batch-commit durability]
- [ ] CHK012 Is idempotent upsert-by-`node_id` required so a repeated/retried batch write converges without duplicating a symbol's row (write correctness under retry)? [Consistency, Spec §FR-009, data-model]

## WAL Checkpoint After Bulk Writes

- [x] CHK013 Is WAL-checkpoint behavior after the embed pass's bulk vector writes specified to match the existing `runMaintenance()` (`PRAGMA wal_checkpoint(PASSIVE)` + `PRAGMA optimize`) pattern **and** is the pass's ordering relative to the existing `runMaintenance()` call in `indexAll()`/`sync()` pinned — given the pass is the **largest** bulk write and, if it runs after that call (as the vocab-heal at `src/index.ts:634` does, relying on auto-checkpoint), its writes grow the WAL unboundedly past the 1000-page auto-checkpoint threshold? [Resolved →§FR-030: WAL MUST be checkpointed after the pass's bulk writes via the same `runMaintenance()` (PASSIVE checkpoint + `optimize`) the index/sync already runs, with the pass positioned so its writes are covered — run before the existing post-resolution maintenance step or invoke the same maintenance after — never leaving the largest bulk write un-checkpointed; best-effort/advisory like the existing step; plan Performance Goals mirrors]

## Index-Lock Hold Time & Read Responsiveness (special attention)

- [x] CHK014 Is there a requirement that **MCP daemon reads (queries) stay responsive during a long embed pass** that holds the index lock — i.e., the pass never blocks concurrent readers — with the enabling mechanism named (WAL readers run concurrently with the single writer; "with WAL, reads never block on a writer," `src/db/index.ts`) and preserved by the per-batch-commit strategy (no reader-visible pass-long write transaction)? FR-015a specifies write mutual-exclusion but is silent on read responsiveness. [Resolved →§FR-031(a): pass MUST NOT block concurrent readers — MCP daemon queries stay responsive throughout a long pass — guaranteed by WAL (readers concurrent with the single writer) and preserved by the FR-029 batch-commit strategy (no reader-visible pass-long write transaction); plan Performance Goals mirrors. NOTE: escalated to consensus — the read-non-blocking guarantee depends on the daemon read-path connection model (query-pool worker threads holding independent connections), only partially traced here; overlaps the cross-process lock semantics flagged in data-integrity CHK025]
- [x] CHK015 Because the pass performs network I/O while holding the index lock, is the **index-lock-hold duration bounded and expressed measurably** — a fully-down endpoint aborts within one batch's bounded retry budget (retries × backoff-cap) rather than holding the lock indefinitely, and FR-019a's per-request timeout is explicitly connected to lock-hold time — with the honest acknowledgement that a *slow-but-responding* endpoint on a large full index still scales lock-hold with pass length (the very condition FR-015a's 2-minute stale-reclaim limitation already accepts)? [Resolved →§FR-031(b): index-lock hold the pass contributes is bounded per batch by the per-request timeout (FR-019a) × bounded concurrency × bounded retries — a fully-unreachable endpoint aborts within one batch's retry budget; the slow-but-responding long-index case is explicitly tied to FR-015a's accepted two-minute stale-reclaim limitation and remains bounded (never infinite) by the per-request timeout; plan Performance Goals mirrors. NOTE: escalated to consensus — worst-case lock-hold on a slow endpoint interacts with the cross-process stale-reclaim window (same lock semantics flagged in data-integrity CHK025)]
- [ ] CHK016 Is the accepted FR-015a limitation (the inherited file lock's fixed 2-minute stale-reclaim window, no freshness refresh while held, made likelier by a long successful pass) documented with its bounded worst case — a transient orphan/missing vector healed by the next pass, never corrupted graph structure? [Coverage, Spec §FR-015a]
- [ ] CHK017 Is cross-process write contention during the pass bounded and non-hanging (a losing index-lock acquirer fails fast with a no-op result, not a blocking wait), consistent with the 5 s `busy_timeout` and the FR-015a fail-fast-no-queue rule? [Consistency, Spec §FR-015a, `src/db/index.ts` busy_timeout]

## Bounded Concurrency, Timeout & Backpressure

- [ ] CHK018 Are batch size (16), concurrency (4), and per-request timeout (30,000 ms) specified as bounded, env-overridable, positive-int-validated + clamped tunables? [Completeness, Spec §Assumptions/FR-019a, contract embedding-config, research §D4]
- [ ] CHK019 Is a hostile or typo'd tunable (e.g., an enormous batch size or concurrency) prevented from wedging the index lock via clamping to a safe maximum? [Edge Case, research §D4]
- [x] CHK020 Is it specified that concurrency (4) bounds only **in-flight HTTP requests**, while vector writes are serialized synchronously through the single `node:sqlite` (`DatabaseSync`) connection (no concurrent-writer contention on `node_vectors`) — so the concurrency knob can never introduce a DB write race? [Resolved →§FR-029: concurrency governs only in-flight endpoint requests; each completed batch's vectors are written synchronously through the single `node:sqlite` (`DatabaseSync`) connection, so there is no concurrent-writer contention on `node_vectors` and no write race the concurrency knob could introduce; node-vectors-schema contract note mirrors]
- [ ] CHK021 Is backoff bounded (base 1,000 ms ×2, full jitter, ~8 s per-delay cap, 3 retries/batch, `Retry-After` honored capped ~30 s) so retry wall-clock inside the lock is bounded? [Clarity, Spec §Assumptions, research §D5, contract embedding-config]
