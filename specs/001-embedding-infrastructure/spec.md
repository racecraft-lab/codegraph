# Feature Specification: Embedding Infrastructure & Endpoint Provider (SPEC-001)

**Feature Branch**: `001-embedding-infrastructure`

**Created**: 2026-07-04

**Status**: Draft

**Input**: User description: "Build the embedding substrate for CodeGraph: every indexed declaration-level symbol gets a vector computed through a user-configured OpenAI-compatible embedding endpoint, incrementally and resiliently, with the feature fully dormant when unconfigured (zero behavior change, zero network traffic). Semantic retrieval (SPEC-003) and downstream intelligence features (SPEC-011 labels, SPEC-019 wiki) consume the provider interface and the persisted vectors."

## User Scenarios & Testing *(mandatory)*

<!--
  Slice A = P1 (configure + full-index embedding) is a viable MVP on its own.
  Slice B = P2 + P3 (incremental freshness, backfill, resilience) builds on Slice A.

### User Story 1 - Configure and embed on index (Slice A) (Priority: P1)

A user who runs a local or remote OpenAI-compatible embedding endpoint (Ollama, LM Studio, vLLM, a hosted OpenAI-compatible API, etc.) wants semantic vectors over their code. They set the endpoint URL and the model name, run a full index, and every declaration-level symbol comes out with a persisted vector. They confirm coverage by running the status command, which shows the endpoint/backend, the active model, the vector dimension, and 100% coverage.

**Why this priority**: This is the substrate the entire Intelligence Platform stands on — no vectors, no semantic retrieval (SPEC-003), no vector-consuming intelligence features (SPEC-011 labels, SPEC-019 wiki). It is the smallest slice that delivers a complete, observable capability: a configured user gets fully-populated vectors and can see that they are complete. Nothing downstream can start until this exists.

**Independent Test**: Configure the endpoint URL and model, run a full index against a real project, and verify via the status command that coverage over embeddable symbols is 100%, that the reported model and dimension match the configuration/endpoint, and that a spot-checked symbol has a stored vector of the expected dimension. Delivers value with no dependency on Slice B.

**Acceptance Scenarios**:

1. **Given** an endpoint URL and model are configured and a project has never been indexed, **When** the user runs a full index, **Then** every declaration-kind symbol has a persisted vector and status reports 100% coverage, the active model, and the vector dimension.
2. **Given** neither an endpoint URL nor a model is configured (fully unconfigured), **When** the user runs a full index, **Then** no embedding network request is made, no vectors are written, index results are identical to today, and status indicates the embedding feature is dormant (not an error). *(A half-configuration — exactly one of the two set — is a distinct case handled by FR-001a: feature off, but an actionable error names the missing variable.)*
3. **Given** an endpoint that requires no API key (a keyless local endpoint), **When** the user configures only URL and model and runs an index, **Then** embedding succeeds without an API key being set.
4. **Given** the dimension is not explicitly configured, **When** the first batch of vectors returns, **Then** the system infers the dimension from that batch, persists it, and reports it in status.

### User Story 2 - Incremental freshness on edit (Slice B) (Priority: P2)

A user actively editing code wants vectors to stay fresh without paying to re-embed the whole project on every change. Each sync — whether run explicitly from the CLI or triggered automatically by the background daemon's file watcher — re-embeds only the symbols whose embedding input genuinely changed and removes vectors for symbols that were deleted, keeping the vector layer current at negligible endpoint cost.

**Why this priority**: Freshness is what makes the feature usable day-to-day rather than a one-time batch job. It depends on Slice A (there must be vectors to keep fresh), so it is P2. Without it, vectors drift stale as code changes and the user must re-index fully to recover accuracy.

**Independent Test**: With a fully-embedded project (Slice A complete), edit one symbol's body, delete another symbol, and run a sync. Verify that exactly the edited symbol is re-embedded, the deleted symbol's vector is gone, and untouched symbols are not re-embedded (e.g., by observing that the number of embedding requests matches the number of changed symbols, not the total).

**Acceptance Scenarios**:

1. **Given** a fully-embedded project, **When** the user edits one symbol's signature or body and runs a sync, **Then** only that symbol (and any others whose embedding input changed) is re-embedded and all other vectors are left untouched.
2. **Given** a fully-embedded project, **When** the user deletes a symbol and runs a sync, **Then** the vector for the removed symbol is deleted.
3. **Given** a fully-embedded project, **When** a rename or move occurs that does not change a symbol's embedding input, **Then** that symbol is not re-embedded.
4. **Given** the background daemon is watching the project, **When** a watched file changes and triggers an automatic sync, **Then** the embedding pass runs in that daemon sync exactly as it would in a CLI sync.

---

### User Story 3 - Late configuration and endpoint resilience (Slice B) (Priority: P3)

A user who indexed their project before configuring an endpoint later sets the URL and model; a plain sync backfills every missing vector with no special command to learn. Separately, if the endpoint goes down partway through a pass, the index/sync still completes successfully and the pass simply resumes from where it stopped on the next run.

**Why this priority**: This closes the two failure/latecomer paths that would otherwise strand users — configuring after the fact, and transient endpoint outages. It depends on the incremental machinery from Story 2 (backfill and resume are both "embed only what's missing/stale"), so it is P3. It is the resilience polish that makes the feature safe to leave on.

**Independent Test**: (a) Index a project with no endpoint configured (zero vectors), then configure the endpoint and run a single plain sync; verify coverage reaches 100% with no new command used. (b) Start a pass, make the endpoint unreachable partway through, and confirm the operation still reports success with partial vectors written; restore the endpoint, run again, and confirm coverage completes.

**Acceptance Scenarios**:

1. **Given** a project indexed before any endpoint was configured (zero vectors), **When** the user configures the endpoint and runs a plain sync, **Then** all missing vectors are backfilled and coverage reaches 100% — with no new or special command required.
2. **Given** an embedding pass is in progress, **When** the endpoint becomes unreachable, **Then** the system retries a bounded number of times, aborts the pass cleanly leaving already-written vectors in place, and the enclosing index/sync operation still reports success.
3. **Given** a pass previously aborted with some vectors missing, **When** the user runs the next index or sync with the endpoint reachable, **Then** the pass resumes and embeds the still-missing and still-stale symbols with no manual intervention.
4. **Given** the endpoint returns a vector whose dimension conflicts with the enforced dimension, **When** the pass runs, **Then** the system surfaces an actionable error naming `CODEGRAPH_EMBEDDING_DIMS`, treats the pass as failed, and the enclosing operation still succeeds.

### Edge Cases

- **Only one of URL/model set (half-configuration)**: the feature stays **off** (zero network requests, zero writes, and the index/sync still succeeds), but — because setting one variable signals intent to enable — the system surfaces an **actionable config error that names the missing variable** (`CODEGRAPH_EMBEDDING_MODEL` when URL is set without MODEL; `CODEGRAPH_EMBEDDING_URL` symmetrically), rather than the silent, neutral dormancy of the fully-unconfigured (neither-set) case (FR-001a). Both variables are required to activate.
- **Endpoint requires a key but none is set (or the key is wrong)**: the resulting HTTP 401/403 auth failure is handled as a **non-retryable** endpoint failure — fast advisory abort without exhausting the retry budget (FR-003/FR-019) — never as a fatal error to the index/sync. A missing or wrong static bearer key rejects identically on every attempt, so retrying only wastes wall-clock inside the index lock.
- **Active model changed between runs**: every stored row whose model no longer matches the active model is treated as stale and re-embedded (replaced), so switching models converges to a single-model store.
- **Dimension mismatch mid-run**: an endpoint returning a different dimension than the persisted/enforced one raises an actionable error naming `CODEGRAPH_EMBEDDING_DIMS` and aborts the pass advisorily.
- **Process killed mid-pass**: because progress is tracked by which nodes still lack a current vector, the next run resumes automatically — there is no separate resume checkpoint to corrupt.
- **Symbol unchanged but moved (start-line shift)**: node identity is a deterministic function of file path, kind, name, and **start line**, so a symbol whose body is byte-identical but whose start line shifts (e.g., a line inserted above it) gets a **new** node id while its old id disappears from the graph. This is not treated as "unchanged": the old-id vector becomes an orphan and MUST be removed by the FR-017 reconciliation (never silently retained), and the symbol is re-embedded under its new id. A single insertion near the top of a file can therefore re-embed every symbol below it — an accepted cost of line-based identity; the invariant that MUST hold is that no orphan is left behind and coverage still reaches 100%.
- **Empty project or project with no embeddable symbols**: no embedding requests are made and coverage is trivially complete.
- **Very large project**: symbols are embedded in bounded batches; the abort/resume design keeps a failed pass from losing prior progress.
- **Endpoint slow, timing out, or hanging (accepts the connection but never responds)**: each request carries a per-request `AbortSignal.timeout` (default 30,000 ms, override `CODEGRAPH_EMBEDDING_TIMEOUT_MS`), so an endpoint that hangs and never sends a byte is converted into a bounded timeout failure and handled on the bounded-retry-then-advisory-abort path (FR-019/FR-019a). This is deliberately distinct from an *unreachable* endpoint (connection refused, covered by US3 AS2): a hang produces no error on its own, so the per-request timeout is what guarantees the pass — and therefore the enclosing index/sync — never hangs indefinitely.
- **Non-retryable endpoint response (a 4xx other than 429)**: a client error such as HTTP 400/404/422 (a malformed request, or a wrong endpoint path/URL) cannot succeed on retry, so it aborts the pass **fast and advisorily — without exhausting the retry budget** — while the enclosing index/sync still succeeds; only 5xx/429/timeout/network errors consume the bounded-retry budget (FR-019, following the standard client-vs-server error split). Auth **401/403 is included in this fast-abort set** (not the retry bucket): a missing or wrong key rejects identically on every attempt — the same rationale FR-019 already applies to 400/404/422. (Consensus: 3/3 — reference SDKs exclude 401/403 from retry by construction, and retrying auth failures risks IP-level throttling.)
- **Malformed or mismatched endpoint response**: a success (HTTP 200) with a non-JSON/truncated body, a missing embeddings array, or an embedding count that does not match the batch input size is treated as an endpoint failure (retried then advisorily aborted) and is never persisted — vectors are never silently misaligned to the wrong symbol (FR-021a).

## Clarifications

### Session 2026-07-04 (Session 1 — Node identity & vector lifecycle)

- Q: Are symbol identities stable across re-index; is a vector-reuse cache needed? → A: Identities are deterministic text ids (derived from file path, kind, name, and start line — stable for unchanged content); no reuse cache. The full CLI re-index recreates the database and re-embeds by design; sync and library re-index preserve vectors for unchanged files.
- Q: Does the `variable` kind include function locals? → A: No — locals are never graph symbols; symbol selection is a flat kind-membership test (FR-005).
- Q: When a modified file's node rows are deleted and re-inserted, must re-embedding be strictly per-symbol (skip vectors for symbols in the touched file whose identity and input hash are unchanged), or is re-embedding every symbol in the touched file acceptable? → A: Strictly per-symbol. Node identity is a deterministic function of file path, kind, name, and start line, so an unaffected symbol regenerates the identical node id on re-extraction; the vector table carries no cascading delete from the node row, so its vector survives the file's node delete-and-reinsert cycle untouched. Vectors for symbols confirmed gone after re-extraction are removed by the explicit reconciliation in FR-017, not as a side effect of the file-level node delete. (Consensus: codebase-analyst + spec-context-analyst, both-agree, Round 1.)

### Session 2026-07-04 (Session 2 — Endpoint client behavior)

- Q: Where are the inferred/enforced dimension and active model persisted so enforcement survives restarts? → A: As project-metadata scalars (written on first successful batch, read at pass start), following the existing index-version stamp pattern; per-row model/dims columns stay self-describing integrity metadata (FR-004 updated).
- Q: Backoff parameters for 5xx/429, given abort-on-exhaustion (Q8)? → A: Base 1,000 ms, ×2 growth, full jitter, ~8 s per-delay cap, 3 retries per batch, honoring `Retry-After` on 429 (capped ~30 s); fixed constants, not env vars.
- Q: Embedding-input truncation cap? → A: Fixed ~6,000-character cap on the composed input, character-based (token counting would need a tokenizer dependency, FR-025); small-context models may still truncate server-side — accepted.
- Q: Defaults for batch size / concurrency / timeout? → A: 16 / 4 / 30,000 ms, all env-overridable with positive-int validation and clamping.

### Session 2026-07-04 (Session 3 — Status surface & slice boundary)

- Q: How is coverage computed given transient orphan vector rows (no-FK design)? → A: Join from live embeddable symbols to vector rows filtered to the active model; "current" = present ∧ model-match (input-hash staleness is a sync-time trigger, not a status-time check). FR-022 updated.
- Q: What does the status embedding section show, and does `--json` get parity? → A: Endpoint (redacted), model, dimension, coverage as `embedded/embeddable (NN%)`; a parallel `embedding` object in `--json` (required — automated probes read machine output).
- Q: What renders for the endpoint line (credential safety)? → A: Scheme + host + port only — userinfo, path, and query stripped; the API key never rendered anywhere (FR-023 extended). Strictest option chosen; security-flagged decision surfaced for operator review.
- Q: What shows when dormant? → A: A neutral (never warning-styled) dormant line naming the two activation variables; if vectors from a prior configured run exist on disk, their model/dims/coverage are also shown labeled "from a previous run" (disk-read only — dormancy preserved).
- Q: Does the progress surface gain an embedding phase? → A: Yes — `embedding` added to the progress phase union with a display label, emitted only when active, progress = embedded ÷ eligible.
- Q: Does the Q10 slice split map onto user stories? → A: Confirmed 1:1 — Slice A = US1 (including the full observability surface: status section, coverage, progress phase); Slice B = US2+US3. The "current = present ∧ model-match" reading removes the only straddle, so Slice A has no dependency on Slice B.

### Functional Requirements

#### Activation and configuration

- **FR-001**: The system MUST treat the embedding feature as active only when BOTH an embedding endpoint URL and an embedding model are configured (`CODEGRAPH_EMBEDDING_URL` and `CODEGRAPH_EMBEDDING_MODEL`). When **neither** is set, the feature MUST remain fully dormant (FR-002). When **exactly one** is set — a half-configuration that signals intent to enable — the feature MUST stay off but MUST surface an actionable configuration error rather than silent dormancy (FR-001a).
- **FR-001a**: When exactly one of `CODEGRAPH_EMBEDDING_URL` / `CODEGRAPH_EMBEDDING_MODEL` is set (a detectable half-configuration), the system MUST surface an **actionable configuration error that names the missing variable** — `CODEGRAPH_EMBEDDING_MODEL` when a URL is set without a model, and `CODEGRAPH_EMBEDDING_URL` symmetrically when a model is set without a URL — so a user who intended to enable embeddings learns exactly which variable is missing. In this half-configured state the feature stays **off**: it makes zero embedding-endpoint network requests and zero `node_vectors` writes, and the enclosing index/sync still completes successfully (the config error is advisory, never fatal). This state is deliberately **distinct** from the fully-unconfigured (neither-set) dormant state of FR-002 — which stays byte-identical to a build without the feature — the actionable error being the only observable difference. This actionable error surfaces on **both** channels, driven by the same single config check: (1) a one-line advisory printed to the invoking `index`/`sync` command's own output (non-fatal — the command still completes and reports success, and never exits non-zero) at the point activation is evaluated, and (2) the `status` embedding section, which renders a distinct misconfigured line — never the neutral dormant line (FR-022) — for as long as the half-configuration persists, mirrored in `--json` (`misconfigured: true` + `missingVariable`). Both surfaces render the identical missing-variable message and MUST NOT introduce network traffic or writes. (Consensus: both-agree — the repo's existing two-surface warn convention, e.g. worktree-mismatch and reindex-recommended states.)
- **FR-002**: When dormant, the system MUST make zero embedding-endpoint network requests and MUST produce index and sync results identical to a build without the feature (zero behavior change).
- **FR-003**: The embedding API key (`CODEGRAPH_EMBEDDING_API_KEY`) MUST be optional so that keyless local endpoints work; when the endpoint requires a key and none is supplied (or an invalid key is supplied), the resulting HTTP 401/403 authentication failure MUST be handled as a **non-retryable** endpoint failure — aborting the pass fast, without exhausting the retry budget, in the same bucket as the other terminal 4xx responses (see FR-019) — and MUST NOT be treated as fatal to the enclosing index/sync.
- **FR-004**: The embedding dimension (`CODEGRAPH_EMBEDDING_DIMS`) MUST be optional; when unset, the system MUST infer it from the first successful batch, persist it (as a project-metadata scalar, alongside the active model identity), and enforce it on all subsequent vectors.

#### Symbol selection

- **FR-005**: The system MUST embed only declaration-kind symbols: `function`, `method`, `class`, `struct`, `interface`, `trait`, `protocol`, `enum`, `type_alias`, `module`, `namespace`, `component`, `route`, plus `constant` and `variable` (the extractor emits these kinds only at file/module scope or as type-member constants — never for function locals, which are not graph symbols — so selection is a flat kind-membership test with no scope predicate).
- **FR-006**: The system MUST NOT embed noise-level kinds: `parameter`, `import`, `export`, `enum_member`, `field`, `property`, and `file`.

#### Embedding input and change detection

- **FR-007**: The system MUST compose a deterministic embedding input per symbol from its name, kind, signature, docstring, and a trimmed source snippet — identical inputs MUST always produce the identical composed text. To make "identical" platform-independent, the composed input MUST be normalized before use: line endings normalized to LF (so CRLF vs LF, i.e. a Windows checkout vs a Unix checkout of the same code, produces byte-identical text) and encoded as UTF-8. Two symbols with semantically identical content MUST compose to byte-identical text regardless of the host OS or the repository's line-ending/checkout settings.
- **FR-008**: The system MUST derive a stable input hash (SHA-256 over the UTF-8 bytes of the normalized composed input) and use that hash to decide whether a symbol's stored vector is out of date. Because the input is normalized (FR-007), the hash for a given symbol MUST be identical across platforms and runs — a re-index of the same code on a different OS (or after a line-ending renormalization) MUST NOT mark unchanged symbols as stale or trigger a spurious whole-project re-embed.

#### Persistence

- **FR-009**: The system MUST persist exactly one vector per embedded symbol, keyed by symbol identity, storing the active model name, the vector dimension, and the input hash alongside the vector.
- **FR-010**: The store MUST hold vectors for exactly one active model; any stored row whose model does not match the active model MUST be treated as stale and replaced on the next pass (no multi-model storage). "Match" is an **exact, case-sensitive string equality** between the row's stored `model` and the persisted active-model identity (`project_metadata.embedding_model`, the same value the coverage join filters on) — the configured `CODEGRAPH_EMBEDDING_MODEL` verbatim, with no normalization, tag-stripping, or aliasing. Consequently, changing the configured model string in any way (including a differing tag or letter case) is intentionally treated as a new model and converges the store to it via re-embed; the requirement's guarantee is only that the store ends every pass holding exactly one model's vectors.
- **FR-011**: Each vector MUST be stored as a compact binary blob of little-endian 32-bit floats, with no native or external storage component (brute-force scan is acceptable in this version; approximate-nearest-neighbor indexing and quantization are out of scope).
- **FR-012**: The schema addition MUST ship as a versioned migration applied in lockstep with the base schema definition, so both freshly-created and upgraded databases converge to the same shape. The migration MUST be **DDL-only** (no data backfill — the table starts empty and later passes populate it) and **idempotent** (creating the table only if absent, so a re-open, or a fresh database that already has the table from the base schema, is a safe no-op). Because two copies of the table definition exist (the base schema for fresh databases and the migration for upgraded ones), their convergence MUST be **verified** — a check MUST assert that a freshly-created database and a database upgraded from the prior schema version yield an **identical `node_vectors` table shape** — so the two definitions cannot silently drift into differently-shaped tables. The migration MUST also apply atomically (leaving no partial-schema state if it fails), consistent with the existing migration runner.

#### Embedding pass behavior

- **FR-013**: The embedding pass MUST run inline, after reference resolution, as part of both the full-index and the sync operations.
- **FR-014**: The embedding pass MUST be advisory: any failure within it MUST NOT fail the enclosing index or sync operation, which MUST still report success ("advisory — never fail an index over it").
- **FR-015**: The embedding pass MUST run in every context where sync runs, including syncs triggered automatically by the background daemon's file watcher.
- **FR-015a**: Because the pass runs inline inside index/sync (FR-013) and also in the daemon-watcher's sync (FR-015), two passes can be requested for the same project at once (for example, a user's CLI `sync` while the background daemon's watcher also syncs — two separate OS processes opening the same project root). The embedding pass MUST execute **within the enclosing index/sync mutual-exclusion** so that concurrent passes can never interleave writes to `node_vectors`, double-write a row, or race the reconciliation delete against an in-flight embed. That mutual-exclusion is the project's existing index lock in its **entirety** — both the in-process operation mutex and the project-scoped, PID-checked file lock that already serializes index/sync across every caller (CLI, the daemon's watcher-triggered and catch-up syncs, git hooks) — not merely within a single process. The pass MUST NOT introduce a second, independent lock; it inherits the one that already serializes index/sync, unmodified in its acquire/release semantics and at its existing scope (the one sanctioned addition is the optional freshness refresh of FR-031(b), which touches only the held lock file's modification time). A losing acquisition attempt is not queued: the entire index/sync operation fails fast with a no-op result, leaving `node_vectors` untouched for that attempt; the daemon watcher's existing retry path and the user's next manual index/sync re-attempt the same work, converging coverage through the existing backfill/resume behavior (FR-018/FR-020) — no new resume or backoff logic for this case. Idempotent per-row upserts are NOT an acceptable substitute for this mutual exclusion: the reconciliation delete racing a concurrent process's insert is a snapshot-consistency problem only real mutual exclusion resolves. *Known pre-existing limitation (accepted)*: the inherited file lock's stale-reclaim window is a fixed two minutes with no freshness refresh while held; a long, successful embedding pass makes it likelier that a still-live holder crosses that window and a second process reclaims the lock. This limitation predates this feature and is accepted — because vectors are derived, replaceable data, the worst observable outcome is a transient orphan or missing vector healed by the next pass, never corrupted graph structure.

#### Incremental freshness

- **FR-016**: On sync, the system MUST re-embed only symbols whose embedding input hash has changed or that have no current vector, leaving unchanged symbols untouched.
- **FR-016a**: The persisted vector for a symbol MUST NOT be deleted as a side effect of that symbol's node row being deleted and re-inserted during re-extraction of its file (no cascading delete from the node row to the vector row) — the vector survives that cycle undisturbed whenever the symbol's node identity (kind, name, start line, file path) and input hash are unchanged. Cleanup of vectors for symbols that no longer exist after re-extraction is performed solely by the explicit reconciliation in FR-017.
- **FR-017**: On sync, the system MUST delete the stored vectors of symbols that no longer exist (the explicit anti-join reconciliation of FR-016a: `node_vectors` rows whose `node_id` is absent from the **complete** live `nodes` set). This reconciliation MUST run on **every vector-preserving pass** — both `sync` and the library in-place re-index that preserves existing vectors (Clarifications Session 1) — not only on the CLI `sync` path, so that no vector-preserving path can leave a silently-orphaned vector behind. The only pass exempt from reconciliation is the full CLI re-index that recreates the database from scratch (a fresh, empty `node_vectors` table cannot hold orphans). The anti-join MUST be evaluated against the whole live node set (never a partial sync scope), so vectors for files untouched by an incremental pass are never falsely deleted.
- **FR-018**: A plain sync MUST backfill vectors for all embeddable symbols that are missing a current vector (for example, a project indexed before the endpoint was configured) without requiring any new or special command.

#### Resilience

- **FR-019**: On an endpoint failure, the system MUST retry a bounded number of times and then abort the pass cleanly, leaving already-written vectors in place, so the enclosing operation still succeeds. Endpoint failures that consume the bounded-retry budget are transient/server-side conditions — connection/network errors, HTTP 5xx, HTTP 429, and per-request **timeouts**. Each request MUST carry a bounded timeout (`AbortSignal.timeout`, default 30,000 ms — see FR-019a) so an endpoint that **hangs** (accepts the connection but never responds) is converted into a timeout failure and handled on this path, rather than stalling the pass. A **non-retryable** response — a client error such as HTTP 400/401/403/404/422 (a malformed request, a missing or invalid credential, or a wrong endpoint path) that cannot succeed on retry — MUST instead abort the pass fast **without exhausting the retry budget**, so a terminal misconfiguration does not burn wall-clock inside the index lock; it is still advisory (the enclosing operation still succeeds). Retryable, budget-consuming conditions remain scoped to transient/server-side signals only: connection/network errors, HTTP 5xx, HTTP 429, and per-request timeouts.
- **FR-019a**: Every endpoint request MUST be bounded by an independent per-request timeout (`AbortSignal.timeout`, default 30,000 ms, overridable via `CODEGRAPH_EMBEDDING_TIMEOUT_MS`, positive-integer validated and clamped). Because the timeout is applied **per request**, a slow endpoint cannot accumulate unbounded wall-clock across a batch or pass. This per-request bound is the guarantee that the embedding pass — and therefore the enclosing index/sync — can never hang indefinitely on an endpoint that neither responds nor errors.
- **FR-020**: After an aborted pass, the next index or sync MUST resume by embedding the still-missing and still-stale symbols, with no manual intervention or separate resume command.
- **FR-021**: When the endpoint returns a vector whose dimension conflicts with the enforced dimension, the system MUST surface an actionable error message that names `CODEGRAPH_EMBEDDING_DIMS` and MUST treat the pass as failed (advisory — the enclosing operation still succeeds).
- **FR-021a**: The system MUST validate each endpoint response before persisting any vector. A success (HTTP 200) whose body is not valid JSON, is truncated, is missing the expected embeddings, or returns an embedding **count that does not match the batch input size** MUST be treated as an endpoint failure (retried then advisorily aborted) and MUST NOT be written. A response that cannot be parsed into exactly one correctly-sized vector per input MUST fail the batch rather than persist a vector — so vectors are never silently misaligned to the wrong symbol, nor stored with a wrong shape, when an endpoint or an intermediary returns a malformed or partial payload.

#### Pass performance, batching, and resource bounds

- **FR-027**: The incremental-cost guarantee (SC-003) is scoped to **endpoint work**: the number of embedding requests and `node_vectors` writes an incremental sync performs MUST be proportional to the changed/missing symbol set, not to repo size. The pass's fixed per-sync overhead — recomposing and re-hashing embeddable symbols to detect `input_hash` staleness (FR-016) and the FR-017 anti-join over the **complete** live-node set — is deliberately **network-free** `O(embeddable-symbols)` local work (an in-memory SHA-256 comparison and one indexed anti-join, no HTTP, no writes for unchanged symbols); it is accepted as cheap relative to endpoint I/O and MUST NOT itself issue any embedding request or vector write. Consequently a single-file daemon-watcher edit (FR-015) costs a handful of endpoint calls proportional to *that file's* changed symbols plus this bounded local scan — never a whole-repo re-embed. (The `O(embeddable-symbols)` staleness scan is deliberately not narrowed to the changed-file scope, because a symbol left stale by a prior aborted pass in an already-re-extracted file must still be re-selected on resume — FR-020 — so correctness requires the wider scan; the accepted cost is bounded local hashing, never network.)
- **FR-028**: The pass MUST select and process eligible symbols in **bounded batches** (streaming), and MUST NOT materialize all embeddable symbols and their composed inputs (each capped at ~6,000 characters) in memory at once, so peak memory stays bounded independent of repo size on a large (10k+-file) full index. This follows the existing batched-selection-to-avoid-OOM precedent already used on the resolution/backfill path.
- **FR-029**: Vector persistence MUST be committed in **batch-sized transactions** — the vectors produced by one completed embedding batch committed together in a single transaction, following the repo's existing bulk-write transaction precedent — rather than one implicit transaction per row (per-row `fsync` churn) **or** one pass-long transaction held open across network I/O (unbounded WAL growth and non-durable partial progress). Because each batch commits before the next begins, already-written vectors are durably on disk when a later batch aborts — the resume/backfill contract (FR-019/FR-020) depends on this batch-commit durability. Bounded concurrency (default 4) governs only **in-flight endpoint requests**; each completed batch's vectors are written **synchronously through the single `node:sqlite` (`DatabaseSync`) connection**, so there is no concurrent-writer contention on `node_vectors` and the concurrency knob can never introduce a DB write race.
- **FR-030**: After the pass's bulk vector writes, the WAL MUST be checkpointed using the same lightweight maintenance the index/sync already runs after its bulk writes (`PRAGMA wal_checkpoint(PASSIVE)` + `PRAGMA optimize`), so a large full-index pass does not grow the WAL unbounded past the auto-checkpoint threshold. The pass MUST be positioned so its writes are covered by that checkpoint — either by running **before** the existing post-resolution maintenance step or by invoking the same maintenance **after** the pass completes — never leaving the largest bulk write of the operation un-checkpointed for the remainder of the run. This checkpoint is best-effort/advisory (swallowed on failure, never load-bearing for correctness), consistent with the existing maintenance step.
- **FR-031**: Because the pass runs inside the index lock (FR-015a) yet performs network I/O, two hold-time properties MUST hold. (a) **Reads stay responsive**: the pass MUST NOT block concurrent readers. The full non-blocking guarantee applies **when the daemon's query pool is attached and healthy** (its default): each pool worker holds its own read connection, and WAL lets those readers run concurrently with the single writer, preserved by the batched-commit strategy of FR-029 (the pass never holds a reader-visible, pass-long write transaction). Two paths are explicitly narrower and share the writer's own thread/connection by existing design: the status query (always main-thread) and any session running without a healthy pool (direct/stdio mode, or pool degraded/disabled) — on those paths "responsive" means bounded-latency, not never-blocking: the pass yields the event loop between its network awaits, and the only synchronous spans a reader can queue behind are one batch's hash/write work (FR-029), never a pass-long span. (b) **Lock-hold is bounded per batch**: the index-lock hold time the pass contributes is bounded by the per-request timeout (FR-019a) × bounded concurrency × bounded retries, so a fully-unreachable endpoint aborts within one batch's bounded retry budget rather than holding the lock indefinitely. A *slow-but-responding* endpoint on a large full index still scales total hold time with the length of the pass; that long-hold case is exactly the condition FR-015a's accepted two-minute stale-reclaim limitation already covers (long multi-minute lock-holds already occur today on large-repo resolution, independent of this feature), and it remains bounded (never infinite) because every individual request is timeout-capped. Because the lock's staleness check is modification-time-based, the pass SHOULD refresh the held lock file's freshness at batch boundaries (mirroring the existing cooperative-yield checkpoint cadence) so a long, healthy pass does not present as stale to a contending process — a minimal, non-mandatory hardening that narrows the FR-015a window without altering the lock's acquire/release semantics.

#### Observability

- **FR-022**: The status command MUST report the embedding endpoint (redacted to scheme + host + port only — never credentials, path, or query), the active model, the vector dimension, and coverage — in both the human-readable output and the machine-readable (`--json`) output. Coverage is the share of embeddable (declaration-kind) symbols that have a current vector, where "current" means a vector row exists for the symbol's identity with the active model — computed by joining from live symbols, so transient orphan vector rows never count (input-hash staleness is a sync-time re-embed trigger, not a status-time check). When the feature is dormant (fully unconfigured — neither activation variable set), status MUST convey that clearly and neutrally without implying an error; if vectors from a previous configured run are present on disk, status additionally reports their model/dimension/coverage labeled as from a previous run (read from disk only — no network). The neutral dormant rendering applies to the fully-unconfigured state; a **half-configuration** (exactly one activation variable set) instead surfaces the actionable missing-variable error of FR-001a, which is not the neutral dormant line.

#### Security and invariants

- **FR-023**: The API key MUST never be persisted to disk, written to any log, or echoed in any error message. Credentials embedded in the endpoint URL (userinfo or query parameters) MUST likewise never be **persisted to disk nor** rendered in any output — like the key, the endpoint URL and any credentials it embeds are never written under `.codegraph/` (neither the `node_vectors` store nor `project_metadata`, which persists only the active model name and vector dimension, FR-004/FR-009; the URL and key are held in memory only), and the endpoint is always displayed redacted to scheme + host + port. This redaction MUST also apply to error and log messages **derived from underlying transport failures**: a raw `fetch`/network error object or an endpoint response body can embed the full URL (including userinfo) or echo request headers, so any such message MUST be re-rendered with the endpoint redacted rather than surfaced verbatim — neither the API key nor URL-embedded credentials may leak through a passed-through raw error on **any** failure path (timeout, network error, 4xx/5xx, dimension conflict, malformed response). If the endpoint URL cannot be parsed for redaction, it MUST render as a safe non-credential placeholder (never the raw string), and an unparseable `CODEGRAPH_EMBEDDING_URL` is treated as an endpoint failure (advisory) rather than crashing the index. The redaction guarantee extends to the error's **entire representation**, not just its top-level message: redaction MUST apply recursively through the error's `cause`/wrapped-error chain and to non-message own-properties (for example, the `.input` property Node attaches to an invalid-URL error, which carries the raw input string verbatim) — in practice the pass MUST fully **replace** an underlying transport/endpoint error with a new, redacted error before it leaves the pass, never merely wrap it, so no downstream generic handler (including a process-level crash logger) can print the original error or its chain. **Provider response-body content is untrusted**: some providers echo real fragments of the submitted key in auth-failure bodies, so response-body text MUST pass the same redaction before any part of it is surfaced. The **outgoing request/init object** (which carries the Authorization header) MUST never appear in any logged, dumped, or error-context representation. These rules apply to every output sink — terminal output, the persisted error log, and the daemon's log — identically whether the pass runs under the CLI or the background daemon's watcher-triggered sync (FR-015).
- **FR-024**: The feature MUST NOT alter graph structure: symbol (node) and relationship (edge) counts MUST remain identical between an index run with the feature active and one without.
- **FR-025**: The feature MUST NOT add any new runtime dependency (it MUST use the platform's built-in HTTP capability) and MUST NOT introduce any new telemetry.
- **FR-025a**: The composed embedding input — which contains the user's own source code — MUST leave the machine **only** as the body of an embeddings request to the user-configured endpoint (`CODEGRAPH_EMBEDDING_URL`), and **only** while the feature is active (FR-001). That request MUST carry only the standard OpenAI-compatible embeddings payload — the configured model name and the batch of composed input strings, plus the `Authorization` bearer header when a key is set — and MUST NOT attach any other field, identifier, file path, project path, machine identifier, or metadata. The feature MUST NOT introduce any secondary destination, default or fallback endpoint, analytics, or phone-home channel for embedding input or endpoint metadata: there is exactly **one** outbound destination — the endpoint the user explicitly configured — and it exists only because the user configured it. This is the affirmative, code-content form of the dormancy guarantee (FR-002) and the no-new-telemetry guarantee (FR-025): no embedding-derived data (endpoint, model, code content, coverage, or the fact that a pass ran) may be added to the existing anonymous usage telemetry either. Beyond the network, the composed input MUST NOT be written to any **local** sink — it appears in no log line, no error message, and no persisted artifact; **only** its SHA-256 `input_hash` (FR-008) is persisted. A failure that references a specific symbol (for example a malformed-response or dimension-conflict error) MUST identify that symbol by its node id, never by echoing its composed input or source text. This local-sink confidentiality is **additional to** FR-023's protection of the outgoing request/init object: it independently forbids logging or persisting the composed input on its own, on both the CLI and the daemon-watcher path (FR-015).
- **FR-026**: The feature MUST NOT modify the retrieval surface used by agents (no change to how queries are answered); vectors are written but not yet consumed in this spec.

### Key Entities *(include if feature involves data)*

- **Symbol vector record**: one row per embedded symbol. Represents the persisted embedding of a declaration-level symbol. Key attributes: symbol identity (primary key — the existing symbol's deterministic **text** identifier, derived from file path, kind, name, and start line, and therefore stable across sync and re-index of unchanged content), the vector (compact binary blob of little-endian 32-bit floats), the active model name, the vector dimension, and the input hash used for change detection. The store holds exactly one active model's vectors. The record's lifecycle is independent of the node row's delete/re-insert cycle during file re-extraction (no cascading delete; cleanup is by explicit reconciliation, FR-016a/FR-017).
- **Embedding input**: the deterministic text composed per symbol from its name, kind, signature, docstring, and trimmed source snippet. It is hashed to the input hash that drives change detection; it is transient (sent to the endpoint) and not itself persisted beyond the hash.
- **Endpoint configuration**: the user-provided activation surface — required endpoint URL and model, optional API key, optional dimension — sourced from the environment. Presence of both URL and model activates the feature; the API key is never persisted.
- **Embedding pass**: the inline, post-resolution reconciliation that, when active, embeds missing/stale symbols, deletes vectors for removed symbols, and (on failure) aborts advisorily and resumes on the next run. It never fails the enclosing index/sync.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: With an endpoint configured, a full index produces a current vector for 100% of embeddable (declaration-kind) symbols, and status reports 100% coverage along with the correct model and dimension.
- **SC-002**: With neither activation variable set (fully unconfigured), indexing and sync produce results identical to a build without the feature — zero vectors written and zero embedding network requests made — with no observable behavior change. (The half-configured case is covered by SC-009, not this criterion.)
- **SC-003**: After editing code and running a sync, the number of symbols re-embedded equals the number whose embedding input actually changed (plus any previously missing), not the project total; vectors for deleted symbols are removed. This proportionality guarantee is on **endpoint work** (embedding requests and vector writes); the network-free per-sync staleness scan and reconciliation anti-join are bounded local work over the embeddable set, not `O(changed)` (FR-027).
- **SC-004**: A project indexed before the endpoint was configured reaches 100% coverage with a single plain sync, using no new or special command.
- **SC-005**: If the endpoint becomes unreachable partway through a pass, the enclosing index/sync operation still completes successfully; after the endpoint is restored, a subsequent run reaches 100% coverage without manual intervention.
- **SC-006**: Symbol (node) and relationship (edge) counts are identical between an index run with the feature active and one without — the feature adds no graph structure and causes no count drift across re-index.
- **SC-007**: The configured API key never appears in any persisted file, log line, or error message across all success and failure paths — including an error's `cause`/wrapped-error chain and non-message properties, and identically whether the pass runs under the CLI or the background daemon's watcher-triggered sync.
- **SC-008**: The feature adds no new runtime dependency and emits no telemetry event.
- **SC-009**: With exactly one of `CODEGRAPH_EMBEDDING_URL` / `CODEGRAPH_EMBEDDING_MODEL` set, the system surfaces an actionable error naming the missing variable, makes zero embedding network requests and writes zero vectors, and the enclosing index/sync still reports success — distinct from the fully-unconfigured state, which produces no such message.
- **SC-010**: The full-index embedding pass has **no fixed wall-clock target** — embedding throughput is endpoint-bound (the configured endpoint's latency), not codegraph-bound — so performance is judged instead by objectively measurable codegraph-side bounds: bounded batch size and concurrency, a per-request timeout, bounded retries, bounded peak memory via batched/streaming selection (FR-028), vector writes committed in batch-sized transactions with a WAL checkpoint after the pass (FR-029/FR-030), reads staying responsive during the pass (FR-031), and an observable `embedding` progress phase. As the failure bound, a fully-unreachable endpoint aborts the pass within one batch's bounded retry budget while the enclosing index/sync still reports success (FR-031/FR-019).
- **SC-011**: Across all success and failure paths, and identically whether the pass runs under the CLI or the background daemon watcher: the **only** network destination for embedding input is the user-configured endpoint — a network capture during a pass shows requests to no other host, and each request body is exactly the standard `{ model, input }` payload with no added metadata — and the composed embedding input (the user's source code) appears in **no** persisted file, log line, or error message, only its `input_hash` being persisted; no embedding-derived data is emitted as telemetry (FR-025a/FR-025/FR-002).

Recorded scoping decisions (interview Q1-Q9 and the roadmap storage decision):

- **Q1 — symbol selection**: Embed **declaration kinds only** — `function`, `method`, `class`, `struct`, `interface`, `trait`, `protocol`, `enum`, `type_alias`, `module`, `namespace`, `component`, `route`, plus top-level `constant`/`variable`. Skip `parameter`, `import`, `export`, `enum_member`, `field`/`property`, `file` (embedding noise-level kinds is a non-goal).
- **Q2 — storage keying**: The vector store is keyed by symbol identity with **one active model**; model and dimension are metadata columns; "a model-mismatched row is stale and re-embeds (replace)." No multi-model storage.
- **Q3 — pass placement**: The embedding pass runs **inline post-resolution** inside the index/sync operations and is **advisory** — "advisory — never fail an index over it."
- **Q4 — coverage of sync contexts**: Embed **everywhere sync runs**, including the background daemon watcher's syncs.
- **Q5 — backfill mechanism**: Backfill happens via the existing `sync` heal path — **no new CLI command** (no `codegraph embed` command).
- **Q6 — activation**: Activation requires **URL + MODEL both set**; the API key is optional to support keyless local endpoints.
- **Q7 — dimension handling**: Dimension is optional — **inferred from the first batch, persisted, then enforced**; a mismatch raises an actionable error naming `CODEGRAPH_EMBEDDING_DIMS`.
- **Q8 — endpoint failure**: On endpoint failure, **abort the pass after bounded retries and resume on the next run**.
- **Q9 — telemetry**: **No new telemetry.**
- **Storage (roadmap decision 2026-07-03)**: Vectors are stored as a **plain binary blob (little-endian 32-bit floats) with brute-force scan** in this version, preserving the zero-native-dependency constraint.
- **Session 1 clarification — symbol identity**: Symbol identifiers are deterministic (derived from file path, kind, name, and start line), so vectors keyed by symbol identity survive sync and library re-index of unchanged files; the full CLI re-index command recreates the database and re-embeds from scratch by design. No input-hash-keyed vector reuse cache is needed.
- **Session 1 clarification — locals**: Function-local variables are never graph symbols, so no scope predicate is required for symbol selection (FR-005).

Reasonable defaults for details the interview did not fix (to be finalized in planning):

- Symbols are embedded in bounded batches with bounded concurrency and a per-request timeout. Defaults (Session 2): **batch size 16, concurrency 4, request timeout 30,000 ms** — all user-overridable via `CODEGRAPH_EMBEDDING_BATCH_SIZE`, `CODEGRAPH_EMBEDDING_CONCURRENCY`, and `CODEGRAPH_EMBEDDING_TIMEOUT_MS` (positive-integer validated and clamped, following the existing worker-pool env-parse precedent).
- "Bounded retries" (Session 2) is exponential backoff with full jitter: base 1,000 ms, ×2 growth, ~8 s per-delay cap, **3 retries per batch** (4 attempts total), honoring a server-provided `Retry-After` on 429 (capped ~30 s). These are fixed constants, not new env vars. The retry budget is deliberately smaller than hosted-API cookbook defaults because one exhausted batch aborts the whole pass (Q8) — a sustained-down endpoint should fail fast rather than burn wall-clock inside the index lock. The observable contract remains only that the pass aborts advisorily rather than hanging or failing the operation.
- The composed embedding input (Session 2) is capped at a fixed **~6,000 characters** (snippet trimmed to fit), character-based rather than token-counted because accurate token counting would require a tokenizer dependency (FR-025). ~6,000 chars ≈ ~1,500–1,700 code tokens — under the 2,048-token default context of common local models (e.g. nomic-embed-text) and far under hosted 8,191-token limits. Models with very small contexts (256–512 tokens) may still truncate server-side — an accepted limitation. The cap is a fixed deterministic constant (FR-007).
- The inferred/enforced dimension and the active model identity (Session 2) are persisted as **project-metadata scalars** (written on the first successful batch, read at pass start as the authoritative enforcement values), following the existing index-version stamp pattern; the per-row model/dimension columns (FR-009) remain self-describing integrity metadata, not the enforcement source of truth.
- The endpoint speaks the standard OpenAI-compatible embeddings request/response shape; no vendor-specific behavior is assumed or required.
- Coverage in status (Session 3) is computed by joining from live embeddable symbols to their vector rows filtered to the active model ("current" = present with matching model); transient orphan rows are structurally excluded from the count. Input-hash staleness is detected at sync time, not status time. Zero embeddable symbols reports as trivially complete.
- The index/sync progress surface (Session 3) gains an `embedding` phase (emitted only when the feature is active — dormancy adds no phase), with progress = embedded-so-far ÷ eligible-to-embed.

Scope boundaries assumed for this spec:

- Search-side consumption of vectors is **out of scope** (SPEC-003); this spec writes vectors but does not query them.
- A bundled local embedding model is **out of scope** (SPEC-002); this spec only talks to a user-configured endpoint.
- **Transport confidentiality is the endpoint operator's responsibility; the endpoint scheme is user-controlled.** CodeGraph transmits the composed input (and, when a key is set, the `Authorization` bearer header) to exactly the URL configured in `CODEGRAPH_EMBEDDING_URL` — it neither upgrades nor downgrades the scheme and adds no TLS of its own. Plaintext-`http` **local** endpoints (e.g. a loopback Ollama/LM Studio, keyless — US1 AS3) are supported by design. For a **remote** endpoint, using `https` — so the code payload and any bearer key are encrypted in transit — is the user's responsibility; CodeGraph never blocks a plaintext-`http` remote URL, but the pass SHOULD emit a **one-line advisory warning** when the configured endpoint is plaintext `http` to a non-loopback host (source code and any bearer key would cross the network in cleartext) — advisory only, same warn-style convention as the FR-001a half-config notice, never failing or altering the operation; loopback/localhost `http` stays warning-free (the designed local case). This follows from treating the endpoint as user infrastructure (the same reason the key is optional and the URL is never validated against an allowlist), and it does not weaken FR-023: whatever the scheme, the key and any URL-embedded credentials are still never persisted, logged, or rendered.
- Approximate-nearest-neighbor indexes and quantization are **out of scope**, deferred until scale demands.
- No change is made under the retrieval/MCP surface; agent-facing query behavior is untouched until SPEC-003.
