# Data-Integrity Checklist: Graph-Aware Rename

**Purpose**: Validate the quality (completeness, clarity, consistency, coverage) of the *data-integrity* requirements — byte-exact span verification, write atomicity & snapshot fidelity, multi-edit / same-file composition and offset arithmetic, targeted re-sync & post-check index consistency (no node/edge explosion), and concurrent-access safety (sync-mutex / file-watcher serialization) — before implementation.
**Created**: 2026-07-11
**Feature**: [spec.md](../spec.md) · plan: [plan.md](../plan.md) · data-model: [data-model.md](../data-model.md) · contracts: [rename-plan.schema.json](../contracts/rename-plan.schema.json)

**Note**: Unit tests for the *requirements*, not the implementation. Each item asks whether the data-integrity contract is well-specified — not whether code behaves. Focus areas (domain prompt): span verification is byte-exact (UTF-16 code units on JS strings end-to-end, no byte mapping; CRLF/encoding drift lands as a stale-span refusal, never corruption); atomicity leaves no partially-renamed workspace, snapshots cover every touched file before the first write, and restore is byte-identical; targeted re-sync covers exactly the touched files, the post-check queries the post-sync graph, and node/edge counts stay stable (no explosion); concurrent file-watcher / daemon sessions see a consistent index afterward (sync mutex honored); with special attention to **multi-occurrence lines and overlapping edit ranges within one file** — ordering and offset arithmetic under multiple edits per file.

## Span Verification & Offset Integrity

- [ ] CHK001 - Is the offset unit for both stored graph spans and language-server rename ranges specified as UTF-16 code units end-to-end, with no byte↔UTF-16 translation performed anywhere in plan or apply? [Clarity, Spec §Assumptions / §Clarifications Session 1, research §Decision 2]
- [ ] CHK002 - Is the graph-path edit-range derivation specified (start point from the edge / unresolved-ref position; end = start + old-name UTF-16 length) and made load-bearing only through the live-byte span-verify that confirms the slice equals the old name? [Completeness, data-model §RenameEdit / research §Decision 8]
- [ ] CHK003 - Is span verification specified as an independent gate beyond the confidence tier — a live-byte mismatch drops the edit regardless of tier — so a same-named *different* symbol can never be mis-edited? [Consistency, Spec §FR-004 final clause / §FR-005 / §FR-016]
- [ ] CHK004 - Is CRLF / encoding drift between the index and the live working tree specified to resolve as a stale-span refusal with zero writes (never a corrupting write)? [Coverage / Edge Case, Spec §FR-016 / §Edge Cases]

## Write Atomicity & Snapshot Integrity

- [ ] CHK005 - Is a pre-write in-memory byte snapshot required for **every** touched file, captured before **any** file is written? [Completeness, Spec §FR-018 / §FR-020]
- [ ] CHK006 - Is rollback specified to restore each touched file **byte-identically** from its snapshot, so the workspace returns to its exact pre-apply bytes? [Completeness, Spec §FR-019 / §FR-020 / §SC-002]
- [x] CHK007 - Is it required that a **successful** write preserve every byte of each touched file **outside the edited spans** exactly — line endings (LF/CRLF), trailing-newline presence, byte-order mark, indentation, and on-disk encoding round-tripped unchanged — so a rename edits only the verified old-name spans and never emits an incidental whole-file reformatting diff? [Resolved → Spec §FR-020 "Byte-preservation outside edited spans": a successful write round-trips every byte outside the span-verified occurrences (line endings, trailing newline, BOM, encoding), operating on whole-file content (never a line-split-then-rejoin), the forward complement of the FR-019 byte-identical restore and FR-012/SC-008; data-model §ApplyOutcome "Write integrity"]
- [ ] CHK008 - Is the atomicity guarantee's scope on success stated honestly — outcome-atomic through verification plus a per-file temp-sibling→atomic-rename, so each file flips atomically while the cross-file write sequence is bounded (and the index view is closed by the mutex-serialized re-sync)? [Clarity, Spec §FR-020]
- [ ] CHK009 - Is it specified that snapshots are held only until the post-check passes, with the hard mid-write process-kill window called out as the accepted v1 durability limitation? [Assumption, Spec §FR-020 / §Edge Cases]

## Multi-Edit Composition & Offset Arithmetic

- [x] CHK010 - Is the order in which multiple edits **within one file are applied to produce the written bytes** specified (descending / right-to-left by range start, so an applied edit cannot invalidate the offsets of edits not yet applied) — binding on the actual write, not only on the FR-027 before/after **preview** composition? [Resolved → Spec §FR-020 "Intra-file edit-application order": a file's edits are applied to the bytes in descending / right-to-left order by `range` start so an applied edit never invalidates an unapplied offset — the FR-027 preview rule made binding on the actual write; data-model §ApplyOutcome "Write integrity"]
- [x] CHK011 - Are **overlapping or duplicate edit ranges within one file** required to be detected and handled deterministically before writing — identical (fully-coincident) ranges deduplicated, genuinely overlapping (partially-coincident) ranges refused as a malformed plan — so a span is never double-applied or partially overwritten? [Resolved → Spec §FR-020 "Overlapping / duplicate edit ranges": identical duplicate ranges are de-duplicated; a genuine partial overlap is only reachable from a misbehaving LSP workspace edit (graph occurrences are disjoint by FR-005) and degrades that rename to the graph path via the FR-003a runtime-failure contract — never a double-applied span — mirroring the LSP non-overlap rule; data-model §ApplyOutcome "Write integrity". Flagged for consensus: degrade-to-graph vs. an explicit new refusal is a design choice — resolved to the FR-003a-consistent degrade, adding no new refusal reason]
- [ ] CHK012 - For **multi-occurrence lines**, is the same-line composition rule specified (each edit sharing a file+start-line carries the identical pre-edit `lineText`; the after-line is composed by applying every such edit to one copy right-to-left by start character, not by treating each edit's `lineText` as an independent before/after pair)? [Consistency, Spec §FR-027 / schema §edit.lineText]
- [ ] CHK013 - Is the deterministic edit ordering (by file path, then range start line, then start character) specified so offset arithmetic and CLI≡MCP parity are reproducible? [Measurability, Spec §FR-027 / data-model §RenamePlan]

## Index Consistency & Re-Sync Integrity

- [ ] CHK014 - Is the targeted re-sync required to use the resolution-complete sync path (never extraction-only file indexing), scoped by construction to the touched files (which cover every graph-tracked reference of the renamed symbol)? [Completeness, Spec §FR-018 / research §Decision 3]
- [ ] CHK015 - Is the post-check required to query the **post-sync** graph — a touched-file-scoped dual assertion (no unresolved reference still carrying the old name; no node named the old name remains in those files), never repo-wide? [Completeness, Spec §FR-018 / §Edge Cases]
- [x] CHK016 - Is **node/edge count stability across a rename + its re-sync (no index explosion)** asserted as a verifiable requirement or success criterion — not left implicit as a plan-level "unaffected by design" claim? [Resolved → Spec §SC-010 (before/after node & edge count probe: no net growth across rename + re-sync) + §FR-018 "No index explosion" clause; plan §Constitution Check Principle V now cites SC-010 instead of an unchecked "by design" claim; data-model §ApplyOutcome "No index explosion"]
- [ ] CHK017 - Is a re-sync that fails or reports no change (genuine index-lock contention) required to be treated as an apply failure that triggers rollback, so the post-check never runs against an un-updated graph? [Coverage, Spec §FR-018 / research §Decision 3]
- [ ] CHK018 - After an unconditional rollback, is the graph required to be re-synced to match the restored files, so the index is not left reflecting the reverted-away rename? [Completeness, Spec §FR-019]

## Concurrent-Access Safety

- [ ] CHK019 - Is the apply's re-sync required to be serialized with the file watcher on the index mutex (one shared `sync()` path under one mutex plus the cross-process file lock)? [Completeness, Spec §FR-018 / research §Decision 3]
- [x] CHK020 - Is the interaction between the apply's own re-sync and a **file-watcher-triggered sync of the apply's own writes** specified, such that a watcher winning the race (re-syncing the touched files first, so the apply's own `sync()` then reports no change) does **not** cause a spurious "no change → rollback" of a successful write? [Resolved → Spec §FR-018 "Watcher serialization (no spurious rollback)": apply serializes the watcher out of its write→re-sync→post-check window (holds the index mutex across it, or suspends the watcher) so apply's own re-sync observes the changes; a watcher-already-synced empty result (`filesChecked > 0`, `filesModified = 0`, distinct from the lock-failure zero-shape) must not trigger a spurious no-change rollback of a successful write; data-model §ApplyOutcome "Concurrency". Flagged for consensus: the watcher-suspend vs mutex-hold mechanism touches the do-not-regress watcher/daemon surface and warrants a concurrency review]
- [ ] CHK021 - Is post-apply consistency for concurrent daemon / multi-agent sessions specified — every session observing the apply sees a consistent index afterward (mutex + cross-process file lock serialize apply's re-sync against watcher-driven syncs)? [Coverage, Spec §FR-018 / §Assumptions, research §Decision 3]
